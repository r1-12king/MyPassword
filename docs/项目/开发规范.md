# 开发规范

本文档用于约束 `MyPassword` 项目的日常开发、测试、打包和发布记录流程。

## 1. 基本原则

- 每次代码开发完成后，必须先完成本地验证，再生成 Release 安装包。
- 每次生成 Release 包时，必须自动递增版本号，禁止手动覆盖旧版本 APK。
- 每次生成 Release 包后，必须同步更新升级日志。
- 所有可交付测试包统一归档到项目根目录的 `releases/` 目录下。

## 2. 开发完成后的标准流程

每次功能开发完成后，按以下顺序执行：

1. 完成功能开发。
2. 完成本地自测。
3. 确认 `flutter analyze` 通过。
4. 执行 Release 打包脚本。
5. 检查 `releases/` 目录中是否生成新的 APK。
6. 检查 `docs/项目/升级日志.md` 是否自动新增本次版本记录。
7. 将自动生成的升级日志内容补充完整。

## 3. Release 打包规则

手动执行 Release 打包：

```bash
flutter build apk --release
```

打包时需要人工完成以下动作：

- 先手动更新 `pubspec.yaml` 版本号。
- 执行 `flutter build apk --release`。
- 将生成的 APK 手动复制到 `releases/` 目录。
- 按以下格式重命名输出文件：

```text
应用名_release_版本号.apk
```

当前项目实际命名规则为：

```text
MyPassword_release_0.1.1.apk
```

说明：
- `应用名` 固定为 `MyPassword`
- `版本号` 取 `pubspec.yaml` 中的语义化版本号，不包含 `+buildNumber`
- 新版本由开发者手动递增，因此必须避免与历史包同名

## 4. 版本号规范

项目版本号定义位于 `pubspec.yaml`：

```yaml
version: 主版本.次版本.修订号+内部构建号
```

例如：

```yaml
version: 0.4.0+4
```

建议策略：
- 每次打包，至少将 `修订号` 加 `1`
- 每次打包，将 `内部构建号` 加 `1`

例如：

- `0.4.0+4` 下一次打包可调整为 `0.4.1+5`
- `0.4.1+5` 下一次打包可调整为 `0.4.2+6`

如果后续需要升级主版本或次版本，应先手动调整 `pubspec.yaml`，再执行打包。

## 5. 升级日志规范

升级日志文件固定为：

```text
docs/项目/升级日志.md
```

每次打包后，开发者需要手动在升级日志顶部新增一个版本区块，包含：

- 版本号
- 发布日期
- 安装包文件名
- 内部版本号
- `新增 / 优化 / 修复 / 已知问题` 模板

脚本自动生成的内容中，默认会写入：

- `新增 / 优化 / 修复 / 已知问题`

开发人员在提交版本前，必须手动补充真实更新内容。

## 6. Release 产物目录

所有历史 Release 包统一保存在：

```text
releases/
```

要求：
- 不允许删除历史测试包，除非明确做归档清理。
- 不允许用新包覆盖旧包。
- 同一版本号只允许有一个 APK 文件。

## 7. 推荐交付检查项

每次准备交付测试前，至少确认以下内容：

- 应用可以正常安装
- 首次创建保险库流程可用
- 主密码解锁流程可用
- 生物识别解锁流程可用
- 条目新增、编辑、删除、搜索可用
- 导入备份、导出备份可用
- WebDAV 连接测试、上传、恢复可用
- 自动同步开关行为符合预期
- `docs/项目/升级日志.md` 已补充完整

## 8. 相关文件

- 升级日志：`docs/项目/升级日志.md`
- 项目版本：`pubspec.yaml`
