# SecureKeyStorage Design Draft

## Goal

This document defines the design of `SecureKeyStorage` for the Flutter password manager MVP.

The purpose of this component is to provide a narrow, explicit interface for storing small pieces of sensitive metadata in platform-backed secure storage.

This component should not become a generic dumping ground for random app values.

## Why This Layer Exists

The app already has:

- `sqflite` for structured local data
- `CryptoService` for key derivation, vault key wrapping, and field encryption

`SecureKeyStorage` exists because some sensitive materials do not belong in SQLite, even in wrapped form, and should instead live in platform-protected storage.

Typical examples:

- biometric unlock envelope
- session restoration helper material
- secure-storage-backed flags tied to biometric unlock capability

## Security Goals

`SecureKeyStorage` should help enforce these rules:

- do not store the master password
- do not store plaintext vault keys
- do not store large application records
- isolate biometric unlock material from ordinary app settings
- allow secure deletion of sensitive local secrets

## Non-Goals

`SecureKeyStorage` is not responsible for:

- encryption algorithm design
- password derivation
- SQLite row persistence
- UI prompts
- navigation decisions
- generic app settings like theme or sort order

## Recommended Usage Boundary

Use `SecureKeyStorage` only for:

- platform-protected secret fragments
- wrapped key material that must not live in SQLite
- biometric unlock envelopes
- short-lived secure tokens related to unlock state

Do not use it for:

- credential records
- search indexes
- CSV imports
- ordinary preferences
- feature flags unrelated to secrets

## Storage Backends

In Flutter, the practical implementation will usually be:

- `flutter_secure_storage` for a first version

Possible future evolution:

- platform channels for finer Android Keystore or iOS Keychain control

Important practical note:

- `flutter_secure_storage` is acceptable for MVP
- if you later need stricter biometric gating per key, you may need platform-specific implementations

## Data Classification

Recommended classification for local secrets:

### Store in SQLite

- vault metadata row
- wrapped vault key derived from master-password flow
- encrypted credential fields

### Store in SecureKeyStorage

- biometric unlock envelope
- biometric-enabled vault access helper material
- secure session restoration hints if you choose to support them

### Never Persist

- plaintext master password
- plaintext vault key
- plaintext credential values

## Interface Design

Keep the interface small and purpose-driven.

Suggested API:

```dart
abstract class SecureKeyStorage {
  Future<void> writeBiometricEnvelope(String value);

  Future<String?> readBiometricEnvelope();

  Future<void> deleteBiometricEnvelope();

  Future<void> writeSessionMaterial(String value);

  Future<String?> readSessionMaterial();

  Future<void> deleteSessionMaterial();

  Future<void> clearAll();
}
```

This is already close to the upper limit of what MVP should expose.

If you want even stricter boundaries, split the interface:

- `BiometricEnvelopeStorage`
- `SessionMaterialStorage`

For MVP, one interface is fine as long as key ownership remains explicit.

## Recommended Stored Items

### 1. `biometric_envelope`

Purpose:

- allows biometric unlock to restore vault access without storing the master password

Requirements:

- must not contain the plaintext master password
- should be invalidated if biometric unlock is disabled
- should be removable independently

Recommended content:

- a platform-protected wrapped vault key, or
- another short unlock envelope created by `CryptoService`

### 2. `session_material`

Purpose:

- optional helper for restoring a short-lived unlocked session after app restart or process recovery

MVP recommendation:

- keep this minimal or skip it entirely
- do not create “remember forever” behavior in MVP

If you keep it:

- make it revocable
- expire it aggressively
- never use it as a replacement for the master password

## Key Naming Convention

Use stable, explicit storage keys.

Example:

```text
secure.biometric_envelope
secure.session_material
```

If you later add multi-vault support:

```text
secure.<vaultId>.biometric_envelope
secure.<vaultId>.session_material
```

Avoid vague keys like:

- `secret`
- `token`
- `data`

Those become unmaintainable quickly.

## Recommended Flutter Implementation Shape

Example implementation:

```dart
import 'package:flutter_secure_storage/flutter_secure_storage.dart';

class SecureKeyStorageImpl implements SecureKeyStorage {
  SecureKeyStorageImpl({
    FlutterSecureStorage? storage,
  }) : _storage = storage ?? const FlutterSecureStorage();

  final FlutterSecureStorage _storage;

  static const _biometricEnvelopeKey = 'secure.biometric_envelope';
  static const _sessionMaterialKey = 'secure.session_material';

  @override
  Future<void> writeBiometricEnvelope(String value) {
    return _storage.write(
      key: _biometricEnvelopeKey,
      value: value,
    );
  }

  @override
  Future<String?> readBiometricEnvelope() {
    return _storage.read(key: _biometricEnvelopeKey);
  }

  @override
  Future<void> deleteBiometricEnvelope() {
    return _storage.delete(key: _biometricEnvelopeKey);
  }

  @override
  Future<void> writeSessionMaterial(String value) {
    return _storage.write(
      key: _sessionMaterialKey,
      value: value,
    );
  }

  @override
  Future<String?> readSessionMaterial() {
    return _storage.read(key: _sessionMaterialKey);
  }

  @override
  Future<void> deleteSessionMaterial() {
    return _storage.delete(key: _sessionMaterialKey);
  }

  @override
  Future<void> clearAll() async {
    await _storage.delete(key: _biometricEnvelopeKey);
    await _storage.delete(key: _sessionMaterialKey);
  }
}
```

## Android and iOS Behavior Notes

Do not assume all platforms behave identically.

### Android

- storage is commonly backed by Keystore-related mechanisms through the plugin
- behavior can vary across OS versions and device vendors
- biometric gating may require native configuration if you need stronger per-item guarantees

### iOS

- storage typically maps to Keychain semantics
- access control options may be more expressive than the default Flutter wrapper exposes

Conclusion:

- for MVP, `flutter_secure_storage` is acceptable
- for higher assurance biometric behavior, plan a native bridge later

## Relationship with Biometric Unlock

`SecureKeyStorage` should not initiate biometric prompts itself.

Recommended flow:

1. `BiometricService` checks capability
2. user passes biometric prompt
3. app reads `biometric_envelope` through `SecureKeyStorage`
4. `CryptoService` consumes the envelope and restores unlocked state

This keeps concerns separate:

- `BiometricService` handles user authentication prompt
- `SecureKeyStorage` handles secure retrieval
- `CryptoService` handles cryptographic restoration

## Relationship with CryptoService

`CryptoService` should define what secure material is needed.

`SecureKeyStorage` should only:

- persist
- read
- delete

It should not:

- transform payloads
- derive keys
- inspect business meaning deeply

That separation avoids logic drift.

## Serialization Format

Stored secure values will usually be strings because that fits `flutter_secure_storage` well.

Recommended formats:

- base64 for binary payloads
- compact JSON only if the envelope contains multiple fields

Examples:

- `base64(wrapped bytes)`
- JSON string containing `version`, `algorithm`, `payload`

For MVP:

- prefer a single encoded string per item
- keep parsing in `CryptoService` or a dedicated codec

## Deletion Rules

Deletion must be explicit and predictable.

Delete `biometric_envelope` when:

- user disables biometric unlock
- master password is changed
- vault is reset
- secure storage corruption is detected

Delete `session_material` when:

- app is manually locked
- auto-lock triggers, if session restoration is not desired
- biometric unlock is disabled and the material depends on it
- vault reset happens

## Failure Handling

Expected failure cases:

- key not found
- secure storage inaccessible
- platform storage corrupted
- value unreadable or malformed

Recommended handling:

- missing value should return `null`, not crash
- malformed value should be treated as invalid and deleted if safe
- repeated storage failures should surface as explicit app errors, not silent fallbacks

## Error Model Suggestion

```dart
class SecureStorageException implements Exception {
  SecureStorageException(this.message);

  final String message;
}

class SecureStorageReadException extends SecureStorageException {
  SecureStorageReadException() : super('Unable to read secure storage');
}

class SecureStorageWriteException extends SecureStorageException {
  SecureStorageWriteException() : super('Unable to write secure storage');
}
```

Repository or service layers can map these into app-friendly failures.

## Testing Strategy

Do not test this only through UI flows.

Add unit tests for:

- writing and reading biometric envelope
- deletion behavior
- missing-value behavior
- malformed-value handling in upper layers

In app tests:

- verify disabling biometrics removes the stored envelope
- verify vault reset clears secure storage

## What Not To Store Here

- raw credential rows
- normal settings
- import history
- the master password
- plaintext vault key
- long-lived plaintext session secrets

If a value grows into a complex object graph, it probably belongs elsewhere.

## Suggested Class Split

For MVP:

- `secure_key_storage.dart`
- `secure_key_storage_impl.dart`

If complexity grows:

- `secure_storage_keys.dart`
- `secure_storage_codec.dart`

## Recommended MVP Decision

For the first working version:

- use `flutter_secure_storage`
- store only `biometric_envelope`
- skip persistent session restoration
- clear secure storage when vault is reset or biometric unlock is turned off

This is much safer than trying to be clever with background session recovery too early.

## Example Reduced MVP Interface

If you want the strictest MVP API, this is enough:

```dart
abstract class SecureKeyStorage {
  Future<void> writeBiometricEnvelope(String value);

  Future<String?> readBiometricEnvelope();

  Future<void> deleteBiometricEnvelope();

  Future<void> clearAll();
}
```

That is probably the best first version.
