# Local Database Schema Draft

## Goal

This document defines a practical local database schema for the Android password manager MVP.

The design goals are:

- Support the agreed MVP features cleanly
- Keep sensitive data encrypted at rest
- Leave room for later features like TOTP, autofill matching, sync, and passkeys
- Stay simple enough to implement with Room on top of SQLite or SQLCipher

This schema is intentionally local-first. It does not assume cloud sync in the first release.

## Storage Strategy

Recommended stack:

- `Room` for ORM and migrations
- `SQLite` for local storage
- Prefer `SQLCipher` if you want database-level encryption in addition to field-level protection
- Android `Keystore` for wrapping locally stored encryption material

Recommended security model:

- The vault is unlocked with a `master password`
- The master password is never stored directly
- Use `Argon2id` to derive a key from the master password
- Use that derived key to unwrap a vault data key
- Use the vault data key to encrypt sensitive record fields

Important practical point:

- Do not store plaintext passwords, notes, TOTP secrets, or backup secrets in the database
- Prefer encrypting sensitive fields before Room writes them
- Non-sensitive UI helper fields can remain plaintext if needed for list rendering and search

## Encryption Boundary

For MVP, split fields into two groups:

Plaintext or lower-sensitivity metadata:

- Record ID
- Created time
- Updated time
- Favorite flag
- Soft delete flag
- Display title if you accept search convenience over strict zero-knowledge local privacy
- Normalized domain for local matching

Encrypted sensitive fields:

- Username
- Password
- Notes
- TOTP secret
- Card numbers
- Identity numbers
- Recovery codes

Two viable approaches:

1. `Balanced MVP approach`
- Keep `title` and `domain` searchable in plaintext
- Encrypt the rest
- Better UX and simpler local search

2. `Stricter privacy approach`
- Encrypt nearly everything
- Add separate searchable indexes or blind indexes later
- Stronger local privacy, but much more complex

For MVP, I recommend the balanced approach.

## Entity Overview

Recommended MVP tables:

- `vault_meta`
- `credentials`
- `credential_tags`
- `tags`
- `import_history`
- `app_settings`

Optional but useful early:

- `credential_usage`
- `autofill_match`

Later-phase tables:

- `totp_secrets`
- `attachments`
- `sync_change_log`
- `passkeys`

## Table Design

### 1. `vault_meta`

Purpose:

- Stores vault-level configuration and key-wrapping metadata
- There should be only one logical row for MVP

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | Usually a fixed value like `default_vault` |
| `vault_name` | TEXT | Optional display name |
| `master_password_kdf` | TEXT | Example: `argon2id` |
| `kdf_memory_kb` | INTEGER | KDF parameter |
| `kdf_iterations` | INTEGER | KDF parameter |
| `kdf_parallelism` | INTEGER | KDF parameter |
| `salt_base64` | TEXT | Salt used for master-password derivation |
| `wrapped_vault_key_base64` | TEXT | Vault key encrypted using key derived from master password |
| `keystore_alias` | TEXT | Alias for Android Keystore material if used |
| `vault_version` | INTEGER | Schema or vault format version |
| `created_at` | INTEGER | Unix epoch millis |
| `updated_at` | INTEGER | Unix epoch millis |

Notes:

- Do not store the master password or its reversible form
- The wrapped vault key is the critical material here

### 2. `credentials`

Purpose:

- Main vault item table for username/password records
- This is the core MVP entity

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | UUID |
| `title` | TEXT NOT NULL | Display name, optionally plaintext for search |
| `username_ciphertext` | BLOB | Encrypted username |
| `password_ciphertext` | BLOB | Encrypted password |
| `notes_ciphertext` | BLOB | Encrypted notes |
| `website_url` | TEXT | Original URL |
| `website_domain` | TEXT | Normalized domain for matching and search |
| `favorite` | INTEGER NOT NULL DEFAULT 0 | Boolean as 0/1 |
| `category` | TEXT | Optional simple category |
| `created_at` | INTEGER NOT NULL | Unix epoch millis |
| `updated_at` | INTEGER NOT NULL | Unix epoch millis |
| `last_used_at` | INTEGER | For recent items |
| `deleted_at` | INTEGER | Null means active; later helps sync and soft delete |
| `version` | INTEGER NOT NULL DEFAULT 1 | Useful for future sync/conflict handling |

Encryption notes:

- `username_ciphertext`, `password_ciphertext`, and `notes_ciphertext` should contain encrypted payloads
- The encrypted payload can include nonce, ciphertext, and authentication tag in a compact binary envelope

Recommended indexes:

- Index on `title`
- Index on `website_domain`
- Index on `favorite`
- Index on `updated_at`
- Partial filtering in queries for `deleted_at IS NULL`

### 3. `tags`

Purpose:

- Store reusable labels for credentials

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | UUID |
| `name` | TEXT NOT NULL UNIQUE | Tag text |
| `created_at` | INTEGER NOT NULL | Unix epoch millis |

MVP note:

- If you want to move faster, tags can be postponed and replaced with a simple `category` field only

### 4. `credential_tags`

Purpose:

- Many-to-many join between credentials and tags

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `credential_id` | TEXT NOT NULL | FK to `credentials.id` |
| `tag_id` | TEXT NOT NULL | FK to `tags.id` |
| `created_at` | INTEGER NOT NULL | Unix epoch millis |

Primary key:

- Composite primary key on `credential_id`, `tag_id`

### 5. `import_history`

Purpose:

- Track imports for debugging, support, and dedup assistance

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | UUID |
| `source_type` | TEXT NOT NULL | Example: `chrome_csv`, `bitwarden_csv`, `backup_file` |
| `source_name` | TEXT | Original filename or display name |
| `imported_count` | INTEGER NOT NULL | Number of inserted records |
| `skipped_count` | INTEGER NOT NULL DEFAULT 0 | Number of skipped records |
| `failed_count` | INTEGER NOT NULL DEFAULT 0 | Number of failed rows |
| `started_at` | INTEGER NOT NULL | Unix epoch millis |
| `completed_at` | INTEGER | Unix epoch millis |
| `status` | TEXT NOT NULL | Example: `running`, `completed`, `failed` |

### 6. `app_settings`

Purpose:

- Store local device settings that are not part of the encrypted vault content

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | Usually one row, for example `local_settings` |
| `biometric_enabled` | INTEGER NOT NULL DEFAULT 0 | Boolean as 0/1 |
| `auto_lock_seconds` | INTEGER NOT NULL DEFAULT 60 | Auto-lock timeout |
| `screenshot_protection_enabled` | INTEGER NOT NULL DEFAULT 1 | Boolean as 0/1 |
| `clipboard_clear_seconds` | INTEGER NOT NULL DEFAULT 30 | Clipboard clear timeout |
| `theme_mode` | TEXT | Optional |
| `updated_at` | INTEGER NOT NULL | Unix epoch millis |

Note:

- This table can also live in DataStore instead of SQLite
- If you want fewer moving parts, keep these settings in DataStore and leave them out of the Room schema

## Useful Optional MVP Tables

### 7. `credential_usage`

Purpose:

- Record recent usage to support “recent items”

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | UUID |
| `credential_id` | TEXT NOT NULL | FK to `credentials.id` |
| `used_at` | INTEGER NOT NULL | Unix epoch millis |
| `action_type` | TEXT NOT NULL | Example: `viewed`, `copied_username`, `copied_password`, `opened_url` |

MVP note:

- If this feels too heavy, just update `credentials.last_used_at`

### 8. `autofill_match`

Purpose:

- Store site and app matching metadata for future Android Autofill support

Suggested columns:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | TEXT PRIMARY KEY | UUID |
| `credential_id` | TEXT NOT NULL | FK to `credentials.id` |
| `match_type` | TEXT NOT NULL | Example: `domain`, `package_name`, `web_origin` |
| `match_value` | TEXT NOT NULL | Normalized match target |
| `is_primary` | INTEGER NOT NULL DEFAULT 1 | Boolean as 0/1 |
| `created_at` | INTEGER NOT NULL | Unix epoch millis |

Why this helps:

- It prevents overloading the `credentials` table once autofill rules become more complex

## Later-Phase Tables

These should not block MVP, but the current schema should leave room for them.

### `totp_secrets`

Use this if you later want TOTP stored separately from credentials.

Suggested columns:

- `id`
- `credential_id`
- `issuer`
- `account_name`
- `secret_ciphertext`
- `period`
- `digits`
- `algorithm`
- `created_at`
- `updated_at`

For MVP-plus, you can also store TOTP directly inside `credentials` as encrypted data if you want fewer tables.

### `sync_change_log`

Use this when cloud sync arrives.

Suggested columns:

- `id`
- `entity_type`
- `entity_id`
- `change_type`
- `changed_at`
- `sync_state`
- `device_id`

### `attachments`

For storing encrypted file metadata or references later.

### `passkeys`

For future WebAuthn/passkey metadata if your product evolves there.

## Recommended SQL Shape

Illustrative SQL:

```sql
CREATE TABLE vault_meta (
    id TEXT PRIMARY KEY NOT NULL,
    vault_name TEXT,
    master_password_kdf TEXT NOT NULL,
    kdf_memory_kb INTEGER NOT NULL,
    kdf_iterations INTEGER NOT NULL,
    kdf_parallelism INTEGER NOT NULL,
    salt_base64 TEXT NOT NULL,
    wrapped_vault_key_base64 TEXT NOT NULL,
    keystore_alias TEXT,
    vault_version INTEGER NOT NULL,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

CREATE TABLE credentials (
    id TEXT PRIMARY KEY NOT NULL,
    title TEXT NOT NULL,
    username_ciphertext BLOB,
    password_ciphertext BLOB NOT NULL,
    notes_ciphertext BLOB,
    website_url TEXT,
    website_domain TEXT,
    favorite INTEGER NOT NULL DEFAULT 0,
    category TEXT,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    last_used_at INTEGER,
    deleted_at INTEGER,
    version INTEGER NOT NULL DEFAULT 1
);

CREATE INDEX index_credentials_title ON credentials(title);
CREATE INDEX index_credentials_website_domain ON credentials(website_domain);
CREATE INDEX index_credentials_favorite ON credentials(favorite);
CREATE INDEX index_credentials_updated_at ON credentials(updated_at);

CREATE TABLE tags (
    id TEXT PRIMARY KEY NOT NULL,
    name TEXT NOT NULL UNIQUE,
    created_at INTEGER NOT NULL
);

CREATE TABLE credential_tags (
    credential_id TEXT NOT NULL,
    tag_id TEXT NOT NULL,
    created_at INTEGER NOT NULL,
    PRIMARY KEY (credential_id, tag_id),
    FOREIGN KEY (credential_id) REFERENCES credentials(id) ON DELETE CASCADE,
    FOREIGN KEY (tag_id) REFERENCES tags(id) ON DELETE CASCADE
);

CREATE TABLE import_history (
    id TEXT PRIMARY KEY NOT NULL,
    source_type TEXT NOT NULL,
    source_name TEXT,
    imported_count INTEGER NOT NULL,
    skipped_count INTEGER NOT NULL DEFAULT 0,
    failed_count INTEGER NOT NULL DEFAULT 0,
    started_at INTEGER NOT NULL,
    completed_at INTEGER,
    status TEXT NOT NULL
);
```

## Room Entity Suggestion

Recommended Room entities for MVP:

- `VaultMetaEntity`
- `CredentialEntity`
- `TagEntity`
- `CredentialTagCrossRef`
- `ImportHistoryEntity`

If settings move to DataStore, do not create `AppSettingsEntity`.

Example `CredentialEntity` shape:

```kotlin
@Entity(
    tableName = "credentials",
    indices = [
        Index("title"),
        Index("website_domain"),
        Index("favorite"),
        Index("updated_at"),
    ],
)
data class CredentialEntity(
    @PrimaryKey val id: String,
    val title: String,
    @ColumnInfo(name = "username_ciphertext") val usernameCiphertext: ByteArray?,
    @ColumnInfo(name = "password_ciphertext") val passwordCiphertext: ByteArray,
    @ColumnInfo(name = "notes_ciphertext") val notesCiphertext: ByteArray?,
    @ColumnInfo(name = "website_url") val websiteUrl: String?,
    @ColumnInfo(name = "website_domain") val websiteDomain: String?,
    val favorite: Boolean,
    val category: String?,
    @ColumnInfo(name = "created_at") val createdAt: Long,
    @ColumnInfo(name = "updated_at") val updatedAt: Long,
    @ColumnInfo(name = "last_used_at") val lastUsedAt: Long?,
    @ColumnInfo(name = "deleted_at") val deletedAt: Long?,
    val version: Int,
)
```

Implementation note:

- Room stores booleans as integers under the hood
- For encrypted fields, use a repository or type converter layer to keep crypto logic out of UI code

## Search Strategy

For MVP:

- Search on `title`
- Search on `website_domain`
- Optionally search decrypted username in memory after loading a narrow result set

Do not start with:

- Full-text search across encrypted fields
- Complex fuzzy matching
- Blind-index search across all secret fields

That is not worth the complexity in the first release.

## Delete Strategy

Recommended MVP behavior:

- Use soft delete with `deleted_at`
- Hide deleted records from normal queries

Why:

- Makes recovery easier
- Avoids future schema regret when sync arrives
- Supports safer import and dedup workflows

If you want a simpler first cut:

- Hard delete in UI
- Still keep `deleted_at` in schema for forward compatibility

## Migration Guidance

Plan for schema versioning from day one.

Likely early migrations:

- Add TOTP support
- Add autofill matching table
- Split notes or secret blobs into separate tables
- Add sync metadata

Recommended rule:

- Avoid storing business logic assumptions only in code
- Put enough explicit versioning in schema to make migrations deterministic

## MVP Recommendation

If you want the cleanest MVP implementation, use this reduced set:

- `vault_meta`
- `credentials`
- `import_history`

Optional if tags are needed immediately:

- `tags`
- `credential_tags`

Keep settings in `DataStore` for simplicity.

That gives you:

- Secure unlock metadata
- Core password storage
- Import traceability
- Minimal migration burden
