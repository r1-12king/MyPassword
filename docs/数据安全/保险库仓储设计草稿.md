# VaultRepository Design Draft

## Goal

This document defines the design of `VaultRepository` for the Flutter password manager MVP.

`VaultRepository` is the orchestration layer for vault lifecycle operations.

Its job is to coordinate:

- vault existence checks
- vault creation
- vault unlock
- vault lock
- biometric unlock enablement
- vault reset

It should not become:

- a DAO wrapper with no logic
- a crypto implementation
- a UI state manager

## Why This Repository Exists

The password manager has three separate concerns:

1. `sqflite` stores vault metadata and encrypted credential data
2. `CryptoService` handles key derivation, wrapping, unlocking, and field crypto
3. `SecureKeyStorage` persists small secure envelopes for biometric unlock

`VaultRepository` exists to coordinate these pieces into application-level operations.

Without this layer, vault lifecycle logic ends up split across pages, providers, DAOs, and crypto code.

## Core Responsibilities

`VaultRepository` should own:

- determining whether a vault exists
- creating the initial vault
- loading vault metadata from storage
- calling `CryptoService` to unlock
- clearing secure storage when vault state changes
- exposing unlocked and initialized state in a repository-friendly way
- performing destructive vault reset flow safely

`VaultRepository` should not own:

- raw SQL statements
- field-level encryption of credentials
- biometric prompt UI
- route navigation
- screen-specific validation messages

## Primary Collaborators

`VaultRepository` should depend on:

- `VaultMetaDao`
- `CryptoService`
- `SecureKeyStorage`

Optional later dependency:

- `BiometricService` for enablement workflows if you choose to keep that orchestration here

Recommended split:

- `BiometricService` handles authentication prompts
- `VaultRepository` handles the app-level result of enabling or using biometric unlock

## Domain Concepts

Recommended vault state concepts:

- `no vault`
- `vault exists but locked`
- `vault unlocked`

You do not need a full-blown state machine class in MVP, but you do need these states to be explicit in code.

## Public API Suggestion

Recommended repository API:

```dart
abstract class VaultRepository {
  Future<bool> hasVault();

  bool get hasVaultCached;

  bool get isUnlocked;

  Future<void> createVault({
    required String masterPassword,
    bool enableBiometricUnlock = false,
  });

  Future<bool> unlockVault({
    required String masterPassword,
  });

  Future<bool> canUnlockWithBiometrics();

  Future<bool> unlockWithBiometrics();

  Future<void> enableBiometricUnlock();

  Future<void> disableBiometricUnlock();

  Future<void> lockVault();

  Future<void> resetVault();
}
```

Why this API shape works:

- it separates password unlock from biometric unlock
- it makes vault reset explicit
- it exposes only application-level lifecycle operations

## Recommended Internal Fields

Example internal state:

```dart
class VaultRepositoryImpl implements VaultRepository {
  VaultRepositoryImpl({
    required VaultMetaDao vaultMetaDao,
    required CryptoService cryptoService,
    required SecureKeyStorage secureKeyStorage,
  })  : _vaultMetaDao = vaultMetaDao,
        _cryptoService = cryptoService,
        _secureKeyStorage = secureKeyStorage;

  final VaultMetaDao _vaultMetaDao;
  final CryptoService _cryptoService;
  final SecureKeyStorage _secureKeyStorage;

  bool _hasVaultCached = false;
}
```

Do not duplicate too much state here.

Recommended:

- cache whether the vault exists
- derive unlocked state from `CryptoService.isUnlocked`

Avoid:

- keeping another separate “maybe unlocked” flag that can drift

## Vault Creation Flow

Recommended `createVault()` flow:

1. ensure a vault does not already exist
2. call `CryptoService.createVaultSetup(masterPassword)`
3. persist the returned vault metadata via `VaultMetaDao`
4. update local existence cache
5. if biometric unlock should be enabled and a biometric envelope is available:
   - write envelope via `SecureKeyStorage`
6. leave the vault in unlocked state if `CryptoService` already initialized an in-memory session

Important:

- vault creation should be atomic enough that partial creation does not leave the app in a misleading state
- if secure-storage write fails after metadata is saved, that should disable biometric unlock, not fail vault creation entirely

Pseudo flow:

```text
createVault(masterPassword)
  -> CryptoService.createVaultSetup()
  -> VaultMetaDao.insertVaultMeta()
  -> optional SecureKeyStorage.writeBiometricEnvelope()
  -> cache hasVault = true
```

## Unlock Flow

Recommended `unlockVault()` flow:

1. load vault metadata from `VaultMetaDao`
2. if no metadata exists, return false or throw a domain error
3. pass metadata and password to `CryptoService.unlock()`
4. return the unlock result

Pseudo flow:

```text
unlockVault(masterPassword)
  -> VaultMetaDao.getVaultMeta()
  -> CryptoService.unlock(masterPassword, vaultMeta)
  -> true / false
```

Do not let the UI load metadata rows directly.

## Biometric Unlock Flow

Recommended `canUnlockWithBiometrics()` flow:

1. ask `SecureKeyStorage` whether a biometric envelope exists
2. optionally also ask `CryptoService.canUnlockWithBiometricEnvelope()`
3. return true only if the envelope exists and is readable

Recommended `unlockWithBiometrics()` flow:

1. read biometric envelope from `SecureKeyStorage`
2. if missing, return false
3. pass the envelope to `CryptoService` or let `CryptoService` read via a storage abstraction
4. if restore succeeds, return true
5. if restore fails because envelope is corrupted, delete it and return false

Important separation:

- biometric prompt should already have succeeded before this repository method is called

That means a typical app flow is:

1. UI asks `BiometricService.authenticate()`
2. on success, UI or provider calls `VaultRepository.unlockWithBiometrics()`

## Enable Biometric Unlock Flow

Recommended `enableBiometricUnlock()` flow:

1. ensure the vault is already unlocked
2. ask `CryptoService` to create a biometric unlock envelope from the current session
3. persist it via `SecureKeyStorage`

Why unlocked state is required:

- the repository should only enable biometric unlock after it has a valid current vault session

Recommended failure behavior:

- if storing the envelope fails, keep the vault unlocked but report biometric enablement failure

## Disable Biometric Unlock Flow

Recommended `disableBiometricUnlock()` flow:

1. delete biometric envelope from `SecureKeyStorage`
2. do not destroy the vault itself
3. do not affect ordinary master-password unlock

This is an important separation:

- disabling biometrics is not the same as resetting the vault

## Lock Flow

Recommended `lockVault()` flow:

1. call `CryptoService.lock()`
2. delete short-lived session material if you later store any
3. keep biometric envelope unless the user disabled it

Pseudo flow:

```text
lockVault()
  -> CryptoService.lock()
  -> optional SecureKeyStorage.deleteSessionMaterial()
```

## Reset Vault Flow

This is the most destructive operation and should be explicit.

Recommended `resetVault()` flow:

1. call `CryptoService.lock()`
2. clear secure storage through `SecureKeyStorage.clearAll()`
3. delete vault metadata row
4. delete credential rows
5. reset local cache

Pseudo flow:

```text
resetVault()
  -> CryptoService.lock()
  -> SecureKeyStorage.clearAll()
  -> VaultMetaDao.deleteVaultMeta()
  -> CredentialDao.deleteAll()
  -> hasVaultCached = false
```

Important:

- do not partially reset only some components
- if full wipe fails midway, surface the failure clearly so the app can prompt the user again

## DAO Contract

`VaultRepository` should not know SQL details.

Recommended `VaultMetaDao` contract:

```dart
abstract class VaultMetaDao {
  Future<bool> hasVault();

  Future<VaultMetaRecord?> getVaultMeta();

  Future<void> insertVaultMeta(VaultMetaRecord record);

  Future<void> deleteVaultMeta();
}
```

If vault reset also wipes credentials, that should happen through a separate DAO or higher-level wipe coordinator.

For MVP, it is acceptable for `VaultRepository` to also depend on:

- `CredentialDao`

but keep that dependency limited to reset or destructive maintenance flows.

## Error Model

Recommended domain failures:

```dart
class VaultException implements Exception {
  VaultException(this.message);

  final String message;
}

class VaultAlreadyExistsException extends VaultException {
  VaultAlreadyExistsException() : super('Vault already exists');
}

class VaultNotFoundException extends VaultException {
  VaultNotFoundException() : super('Vault not found');
}

class VaultUnlockFailedException extends VaultException {
  VaultUnlockFailedException() : super('Unable to unlock vault');
}
```

Guidance:

- repository can map lower-level storage and crypto errors into these domain errors
- UI should not receive raw secure-storage or crypto exceptions directly

## Caching Guidance

For MVP, simple caching is enough.

Recommended:

- keep `_hasVaultCached`
- refresh it from DAO on startup
- update it after create or reset

Do not:

- cache full vault metadata forever without invalidation
- build a complex reactive cache too early

The metadata row is small enough to reload when needed.

## State Exposure Guidance

If you want provider-friendly state later, expose repository state through a separate provider or controller.

Avoid making `VaultRepository` itself become a notifier class in the first version.

Recommended separation:

- repository exposes async methods and simple getters
- provider/controller converts that into app state for UI

## Example Implementation Shape

```dart
class VaultRepositoryImpl implements VaultRepository {
  VaultRepositoryImpl({
    required VaultMetaDao vaultMetaDao,
    required CryptoService cryptoService,
    required SecureKeyStorage secureKeyStorage,
  })  : _vaultMetaDao = vaultMetaDao,
        _cryptoService = cryptoService,
        _secureKeyStorage = secureKeyStorage;

  final VaultMetaDao _vaultMetaDao;
  final CryptoService _cryptoService;
  final SecureKeyStorage _secureKeyStorage;

  bool _hasVaultCached = false;

  @override
  bool get hasVaultCached => _hasVaultCached;

  @override
  bool get isUnlocked => _cryptoService.isUnlocked;

  @override
  Future<bool> hasVault() async {
    _hasVaultCached = await _vaultMetaDao.hasVault();
    return _hasVaultCached;
  }

  @override
  Future<void> createVault({
    required String masterPassword,
    bool enableBiometricUnlock = false,
  }) async {
    if (await _vaultMetaDao.hasVault()) {
      throw VaultAlreadyExistsException();
    }

    final setup = await _cryptoService.createVaultSetup(masterPassword);
    await _vaultMetaDao.insertVaultMeta(setup.vaultMetaRecord);

    _hasVaultCached = true;

    if (enableBiometricUnlock && setup.biometricEnvelope != null) {
      await _secureKeyStorage.writeBiometricEnvelope(
        setup.biometricEnvelope!,
      );
    }
  }

  @override
  Future<bool> unlockVault({
    required String masterPassword,
  }) async {
    final vaultMeta = await _vaultMetaDao.getVaultMeta();
    if (vaultMeta == null) {
      throw VaultNotFoundException();
    }

    return _cryptoService.unlock(
      masterPassword: masterPassword,
      vaultMeta: vaultMeta,
    );
  }

  @override
  Future<bool> canUnlockWithBiometrics() async {
    final envelope = await _secureKeyStorage.readBiometricEnvelope();
    return envelope != null && envelope.isNotEmpty;
  }

  @override
  Future<bool> unlockWithBiometrics() async {
    final envelope = await _secureKeyStorage.readBiometricEnvelope();
    if (envelope == null || envelope.isEmpty) return false;

    return _cryptoService.unlockWithBiometricEnvelope();
  }

  @override
  Future<void> enableBiometricUnlock() async {
    if (!_cryptoService.isUnlocked) {
      throw VaultUnlockFailedException();
    }

    final envelope = await _cryptoService.createBiometricEnvelope();
    await _secureKeyStorage.writeBiometricEnvelope(envelope);
  }

  @override
  Future<void> disableBiometricUnlock() async {
    await _secureKeyStorage.deleteBiometricEnvelope();
  }

  @override
  Future<void> lockVault() async {
    await _cryptoService.lock();
    await _secureKeyStorage.deleteSessionMaterial();
  }

  @override
  Future<void> resetVault() async {
    await _cryptoService.lock();
    await _secureKeyStorage.clearAll();
    await _vaultMetaDao.deleteVaultMeta();
    _hasVaultCached = false;
  }
}
```

## Important Refinement

In the example above, `unlockWithBiometricEnvelope()` still hides one design choice:

- should `CryptoService` read the stored envelope itself
- or should `VaultRepository` pass the envelope in explicitly

Recommended final design:

- pass the envelope explicitly

That keeps `CryptoService` independent from secure storage reads.

Better shape:

```dart
Future<bool> unlockWithBiometricEnvelope(String envelope);
```

Then repository becomes the coordinator and storage boundary remains cleaner.

## Recommended Final Contract Adjustments

If you want the cleanest dependency graph, use this split:

`VaultRepository`:

- reads metadata from DAO
- reads secure envelope from secure storage
- passes both into crypto layer
- handles cache and wipe behavior

`CryptoService`:

- performs all cryptographic transforms
- does not read SQLite
- does not read secure storage directly

That is the better long-term design.

## MVP Implementation Order

Recommended order:

1. `hasVault()`
2. `createVault()`
3. `unlockVault()`
4. `lockVault()`
5. `disableBiometricUnlock()`
6. `enableBiometricUnlock()`
7. `unlockWithBiometrics()`
8. `resetVault()`

## What Not To Do

- do not let pages call DAO and `CryptoService` separately
- do not let `VaultRepository` become a generic database facade
- do not store UI validation state in repository
- do not make biometric unlock silently fall back to insecure session restore
- do not let `resetVault()` ignore secure-storage cleanup
