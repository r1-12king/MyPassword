# CredentialRepository 设计草案

## 1. 文档目标

本文档定义密码管理器 MVP 中 `CredentialRepository` 的职责、接口、数据流和实现边界。

这层的目标不是简单“包一层 DAO”，而是把以下几件事组合成上层可以直接使用的业务能力：

- 凭据增删改查
- 搜索
- 字段加解密
- 数据模型映射
- 软删除与基础排序

`CredentialRepository` 是“凭据业务入口层”。

## 2. 为什么需要这一层

密码管理器里的凭据记录不是普通表单数据。

一条记录通常同时涉及：

- 本地数据库字段
- 密文字段
- 明文字段
- UI 列表展示模型
- UI 详情展示模型

如果没有 `CredentialRepository`，常见问题会马上出现：

- 页面直接查 DAO
- 页面自己解密字段
- 列表和详情使用两套不一致的数据结构
- 搜索逻辑散落在页面和数据库层
- 删除逻辑一部分软删、一部分硬删

所以这层的作用就是把“凭据业务规则”集中起来。

## 3. 设计目标

`CredentialRepository` 需要满足这些目标：

1. `对上层友好`
   - 页面不关心密文格式
   - 页面不关心 SQLite 细节

2. `对安全边界清晰`
   - Repository 可以调用 `CryptoService`
   - 页面不能直接调用底层加密逻辑

3. `对后续扩展友好`
   - 允许后面增加：
   - TOTP
   - 最近使用
   - 标签
   - Autofill match
   - 云同步元数据

4. `MVP 可落地`
   - 先解决核心 CRUD 和搜索
   - 不过度设计复杂查询框架

## 4. 职责边界

### 4.1 应该负责的内容

`CredentialRepository` 应该负责：

- 新增凭据
- 更新凭据
- 获取凭据列表
- 获取凭据详情
- 搜索凭据
- 删除凭据
- 记录最近使用时间
- 将明文输入转换为数据库记录
- 将数据库记录转换为领域模型
- 通过 `CryptoService` 对敏感字段加解密

### 4.2 不应该负责的内容

`CredentialRepository` 不应该负责：

- 主密码验证
- 保险库创建或解锁
- vault 生命周期管理
- 生物识别认证
- SQL 执行细节
- 页面状态管理

这些分别属于：

- `VaultRepository`
- `BiometricService`
- `CredentialDao`
- 页面 Provider / Controller

## 5. 依赖关系

推荐依赖如下：

- `CredentialDao`
- `CryptoService`

可选后续依赖：

- `TagDao`
- `AutofillMatchDao`
- `CredentialUsageDao`

推荐方向：

```text
Feature Page / Provider
  -> CredentialRepository
       -> CredentialDao
       -> CryptoService
```

不允许：

- 页面直接调用 `CredentialDao`
- 页面自己解密数据库字段
- `CredentialDao` 反过来依赖 `CryptoService`

## 6. 数据模型建议

`CredentialRepository` 不应该只暴露数据库 Map，也不应该直接把 DAO 结果原封不动丢给页面。

至少建议区分 3 类模型：

### 6.1 写入输入模型

用于新增和编辑：

```dart
class SaveCredentialInput {
  SaveCredentialInput({
    this.id,
    required this.title,
    this.username,
    required this.password,
    this.websiteUrl,
    this.notes,
    this.category,
    this.favorite = false,
  });

  final String? id;
  final String title;
  final String? username;
  final String password;
  final String? websiteUrl;
  final String? notes;
  final String? category;
  final bool favorite;
}
```

### 6.2 列表展示模型

列表页不需要每次都解密全部详情。

建议列表模型只包含必要字段：

```dart
class CredentialListItem {
  CredentialListItem({
    required this.id,
    required this.title,
    required this.websiteDomain,
    required this.favorite,
    required this.updatedAt,
  });

  final String id;
  final String title;
  final String? websiteDomain;
  final bool favorite;
  final DateTime updatedAt;
}
```

### 6.3 详情展示模型

详情页才解密完整字段：

```dart
class CredentialDetail {
  CredentialDetail({
    required this.id,
    required this.title,

    required this.favorite,

  final String id;
  final String title;

  final String password;
  final String? websiteUrl;
  final String? websiteDomain;
  final String? notes;
  final String? category;
  final bool favorite;
  final DateTime createdAt;
  final DateTime updatedAt;
}
```

## 7. 推荐接口设计

推荐 `CredentialRepository` API：

```dart
abstract class CredentialRepository {
  Future<String> saveCredential(SaveCredentialInput input);

  Future<List<CredentialListItem>> getCredentials({
    String? query,
    bool favoritesOnly = false,
  });

  Future<CredentialDetail?> getCredentialDetail(String id);

  Future<void> deleteCredential(String id);

  Future<void> markCredentialUsed(String id);
}
```

为什么推荐这样设计：

- `saveCredential()` 同时支持新增和更新
- 列表查询支持基础搜索和收藏过滤
- 详情查询单独暴露，避免列表场景无谓解密
- 删除和使用记录是两个明确动作

MVP 先不加分页，先按本地数据量可控来设计。

## 8. 列表与详情的分离原则

这是很关键的一点。

### 列表页

应该尽量只使用明文字段：

- `id`
- `title`
- `website_domain`
- `favorite`
- `updated_at`

原因：

- 列表高频展示
- 不需要每次解密用户名和密码
- 性能和安全都更稳

### 详情页

需要时再解密：

- `username_ciphertext`
- `password_ciphertext`
- `notes_ciphertext`

原则是：

- `只在需要展示敏感内容时解密`

## 9. 保存凭据流程

推荐 `saveCredential()` 逻辑：

### 新增

1. 检查 vault 是否已解锁
2. 标准化字段：
   - 去掉无意义空格
   - 规范化域名
3. 通过 `CryptoService` 加密敏感字段：
   - username
   - password
   - notes
4. 组装数据库记录
5. 调用 `CredentialDao.insert()`
6. 返回凭据 id

### 更新

1. 加载原记录
2. 检查记录存在
3. 使用新字段重新构建记录
4. 对敏感字段重新加密
5. 保持 `created_at`
6. 更新 `updated_at`
7. 调用 `CredentialDao.update()`
8. 返回原 id

推荐流程图：

```text
saveCredential(input)
  -> normalize input
  -> CryptoService.encryptToBytes()
  -> CredentialDao.insert/update()
  -> return credentialId
```

## 10. 获取凭据列表流程

推荐 `getCredentials()` 逻辑：

1. 调用 `CredentialDao.queryActive()`
2. DAO 基于以下字段做过滤：
   - `deleted_at IS NULL`
   - title 模糊搜索
   - website_domain 模糊搜索
   - favorite 过滤
3. Repository 将数据库记录映射为 `CredentialListItem`
4. 返回列表

MVP 不建议在列表查询里：

- 解密所有用户名
- 解密所有密码
- 对密文字段做全文检索

这会明显增加复杂度，也没有必要。

## 11. 获取凭据详情流程

推荐 `getCredentialDetail(id)` 逻辑：

1. 调用 `CredentialDao.getById(id)`
2. 若记录不存在或已删除，返回 `null`
3. 通过 `CryptoService.decryptFromBytes()` 解密：
   - username
   - password
   - notes
4. 映射为 `CredentialDetail`
5. 返回详情模型

流程图：

```text
getCredentialDetail(id)
  -> CredentialDao.getById(id)
  -> CryptoService.decryptFromBytes()
  -> CredentialDetail
```

## 12. 删除策略

MVP 建议采用 `软删除`。

### 推荐做法

- 设置 `deleted_at`
- 更新 `updated_at`
- 正常查询全部过滤 `deleted_at IS NULL`

### 为什么不建议直接硬删

- 后面做同步时更容易扩展
- 导入去重时更灵活
- 用户误删恢复空间更大

### Repository 层行为

`deleteCredential(id)` 应该：

1. 校验记录存在
2. 调用 `CredentialDao.softDelete(id, deletedAt)`

如果你前期非常想快，也可以 UI 上表现成“删除即消失”，但底层仍建议保留 `deleted_at` 字段。

## 13. 搜索策略

MVP 搜索建议只做：

- `title`
- `website_domain`

可选增强：

- 先查询明文字段候选集
- 再在小范围结果里解密 username 进行二次过滤

但 MVP 第一版不建议直接上。

不建议一开始做：

- 密文字段全文索引
- 模糊拼音搜索
- 高级过滤器组合

## 14. 最近使用策略

`markCredentialUsed(id)` 建议只做一件事：

- 更新 `last_used_at`

如果后面你想做更强分析，可以再引入：

- `credential_usage` 表

但 MVP 阶段：

- 更新 `credentials.last_used_at` 已够用

## 15. DAO 契约建议

为了让 `CredentialRepository` 不依赖 SQL 细节，建议 `CredentialDao` 至少提供：

```dart
abstract class CredentialDao {
  Future<void> insert(Map<String, Object?> row);

  Future<void> update(String id, Map<String, Object?> row);

  Future<Map<String, Object?>?> getById(String id);

  Future<List<Map<String, Object?>>> queryActive({
    String? query,
    bool favoritesOnly = false,
  });

  Future<void> softDelete(String id, int deletedAt);

  Future<void> updateLastUsedAt(String id, int timestamp);
}
```

Repository 只关心：

- 插入什么
- 查到什么
- 映射成什么

DAO 负责：

- SQL
- where
- order by
- index 友好查询

## 16. 错误模型建议

推荐定义凭据领域错误：

```dart
class CredentialException implements Exception {
  CredentialException(this.message);

  final String message;
}

class CredentialNotFoundException extends CredentialException {
  CredentialNotFoundException() : super('Credential not found');
}

class CredentialVaultLockedException extends CredentialException {
  CredentialVaultLockedException() : super('Vault is locked');
}

class CredentialSaveFailedException extends CredentialException {
  CredentialSaveFailedException() : super('Unable to save credential');
}
```

建议：

- DAO 错误不要直接暴露给 UI
- Crypto 解密失败不要直接原样抛到页面
- Repository 将它们映射成凭据领域错误

## 17. 推荐实现骨架

```dart
class CredentialRepositoryImpl implements CredentialRepository {
  CredentialRepositoryImpl({
    required CredentialDao credentialDao,
    required CryptoService cryptoService,
  })  : _credentialDao = credentialDao,
        _cryptoService = cryptoService;

  final CredentialDao _credentialDao;
  final CryptoService _cryptoService;

  @override
  Future<String> saveCredential(SaveCredentialInput input) async {
    if (!_cryptoService.isUnlocked) {
      throw CredentialVaultLockedException();
    }

    final now = DateTime.now().millisecondsSinceEpoch;
    final id = input.id ?? const Uuid().v4();

    final row = <String, Object?>{
      'id': id,
      'title': input.title.trim(),
      'username_ciphertext': input.username == null || input.username!.isEmpty
          ? null
          : await _cryptoService.encryptToBytes(input.username!.trim()),
      'password_ciphertext': await _cryptoService.encryptToBytes(input.password),
      'notes_ciphertext': input.notes == null || input.notes!.isEmpty
          ? null
          : await _cryptoService.encryptToBytes(input.notes!.trim()),
      'website_url': input.websiteUrl,
      'website_domain': _normalizeDomain(input.websiteUrl),
      'favorite': input.favorite ? 1 : 0,
      'category': input.category,
      'updated_at': now,
      'deleted_at': null,
      'version': 1,
    };

    if (input.id == null) {
      row['created_at'] = now;
      await _credentialDao.insert(row);
    } else {
      await _credentialDao.update(id, row);
    }

    return id;
  }

  @override
  Future<List<CredentialListItem>> getCredentials({
    String? query,
    bool favoritesOnly = false,
  }) async {
    final rows = await _credentialDao.queryActive(
      query: query,
      favoritesOnly: favoritesOnly,
    );

    return rows.map(_mapListItem).toList();
  }

  @override
  Future<CredentialDetail?> getCredentialDetail(String id) async {
    final row = await _credentialDao.getById(id);
    if (row == null) return null;

    return CredentialDetail(
      id: row['id'] as String,
      title: row['title'] as String,
      username: await _decryptNullable(row['username_ciphertext']),
      password: await _cryptoService.decryptFromBytes(
        (row['password_ciphertext'] as List<Object?>).cast<int>(),
      ),
      websiteUrl: row['website_url'] as String?,
      websiteDomain: row['website_domain'] as String?,
      notes: await _decryptNullable(row['notes_ciphertext']),
      category: row['category'] as String?,
      favorite: (row['favorite'] as int) == 1,
      createdAt: DateTime.fromMillisecondsSinceEpoch(row['created_at'] as int),
      updatedAt: DateTime.fromMillisecondsSinceEpoch(row['updated_at'] as int),
    );
  }

  @override
  Future<void> deleteCredential(String id) async {
    await _credentialDao.softDelete(
      id,
      DateTime.now().millisecondsSinceEpoch,
    );
  }

  @override
  Future<void> markCredentialUsed(String id) {
    return _credentialDao.updateLastUsedAt(
      id,
      DateTime.now().millisecondsSinceEpoch,
    );
  }

  CredentialListItem _mapListItem(Map<String, Object?> row) {
    return CredentialListItem(
      id: row['id'] as String,
      title: row['title'] as String,
      websiteDomain: row['website_domain'] as String?,
      favorite: (row['favorite'] as int) == 1,
      updatedAt: DateTime.fromMillisecondsSinceEpoch(row['updated_at'] as int),
    );
  }

  Future<String?> _decryptNullable(Object? value) async {
    if (value == null) return null;
    return _cryptoService.decryptFromBytes((value as List<Object?>).cast<int>());
  }

  String? _normalizeDomain(String? url) {
    if (url == null || url.trim().isEmpty) return null;
    return Uri.tryParse(url.trim())?.host.toLowerCase();
  }
}
```

## 18. 重要实现细节

### 18.1 新增和更新共用一个接口

这样可以减少页面层分叉逻辑。

### 18.2 详情才解密

这能控制性能与安全边界。

### 18.3 Repository 必须依赖 vault 已解锁

如果 vault 锁着，就不允许做需要解密或加密的业务动作。

### 18.4 列表模型不要携带明文密码

列表模型越轻越好。

## 19. 后续扩展点

这份设计后续可以自然扩展到：

### 标签

新增：

- `TagDao`
- 凭据和标签 join 逻辑

### TOTP

新增：

- `totpSecretCiphertext`
- TOTP service

### Autofill 匹配

新增：

- `AutofillMatchDao`
- 凭据与 app/domain 的匹配元数据

### 云同步

新增：

- `version`
- `sync_state`
- `device_id`

不会推翻当前 CRUD 主线。

## 20. 不应该做的事情

- 不要让页面直接解密数据库字段
- 不要在 DAO 里做加密
- 不要让列表接口返回完整明文详情
- 不要把主密码传进 `CredentialRepository`
- 不要让 Repository 直接操控 UI 状态

## 21. 结论

`CredentialRepository` 在整个架构里的定位应该是：

- 对上：向页面提供“可直接用的凭据业务能力”
- 对下：协调 DAO 和 `CryptoService`
- 对内：统一 CRUD、搜索、软删除、字段映射和解密时机

最重要的设计原则只有一句话：

- `列表尽量轻，详情再解密，安全逻辑不要上浮到 UI`
