# CryptoService Design Draft

## Goal

This document defines the design of `CryptoService` for the Flutter password manager MVP.

The service is responsible for:

- vault creation
- vault unlock
- in-memory session management
- field encryption and decryption
- wrapping and unwrapping the vault key

It is not responsible for:

- database queries
- route navigation
- widget state
- CSV parsing

The design is intentionally MVP-focused and local-first.

## Security Goals

The implementation should enforce these rules:

- the master password is never stored directly
- the vault key is never stored in plaintext on disk
- sensitive field values are encrypted before writing to `sqflite`
- decrypted secrets live in memory only as long as needed
- lock and unlock state is explicit

## Service Boundary

`CryptoService` sits between:

- `VaultRepository`
- `CredentialRepository`
- `SecureKeyStorage`
- future crypto primitives or native platform code

Expected call flow:

- `VaultRepository.createVault()` -> `CryptoService.createVaultSetup()`
- `VaultRepository.unlockVault()` -> `CryptoService.unlock()`
- `CredentialRepository.saveCredential()` -> `CryptoService.encryptToBytes()`
- `CredentialRepository.getCredentialDetail()` -> `CryptoService.decryptFromBytes()`

## Core Model

Recommended key hierarchy:

1. `master password`
   - entered by user
   - never stored

2. `master-derived key`
   - derived from master password with `Argon2id`
   - recreated on unlock
   - used only to wrap or unwrap vault key material

3. `vault key`
   - randomly generated once when the vault is created
   - used for encrypting credential fields
   - should only exist in plaintext in memory while the vault is unlocked

Why this separation matters:

- changing master password later becomes possible
- data encryption stays stable even if unlock credentials change
- field encryption logic is decoupled from password entry flow

## Recommended Algorithms

For MVP:

- KDF: `Argon2id`
- field encryption: `AES-256-GCM` or `XChaCha20-Poly1305`
- random source: cryptographically secure RNG

Pragmatic note:

- `AES-256-GCM` is often easier to support across platforms and libraries
- `XChaCha20-Poly1305` is also strong and nonce-friendly if your chosen library supports it cleanly

Do not:

- invent custom crypto
- use plain SHA hashing as a password KDF
- use a fixed nonce
- reuse one nonce across records

## Responsibilities

`CryptoService` should own:

- salt generation
- KDF parameter selection
- vault key generation
- wrapping and unwrapping vault key
- in-memory unlocked session cache
- field encryption and decryption
- session lock and clear behavior

`CryptoService` should not own:

- direct storage of vault metadata in SQLite
- biometric prompt UI
- route redirects
- app settings persistence

## Public API Suggestion

Suggested high-level Dart API:

```dart
abstract class CryptoService {
  Future<VaultSetupResult> createVaultSetup(String masterPassword);

  Future<bool> unlock({
    required String masterPassword,
    required VaultMetaRecord vaultMeta,
  });

  bool get isUnlocked;

  Future<List<int>> encryptToBytes(String plaintext);

  Future<String> decryptFromBytes(List<int> ciphertext);

  Future<void> lock();

  Future<bool> canUnlockWithBiometricEnvelope();

  Future<bool> unlockWithBiometricEnvelope();
}
```

Two important changes from a naive API:

- `unlock()` should accept `vaultMeta` explicitly rather than reading database state internally
- biometric unlock should be a separate path, not a hidden branch inside password unlock

This keeps the service testable and easier to reason about.

## Supporting Models

Suggested models:

## Vault Creation Flow

Recommended `createVaultSetup()` flow:

1. validate master password strength at a product level
2. generate random salt
3. derive master-derived key with `Argon2id`
4. generate random vault key
5. wrap vault key using the master-derived key
6. serialize vault metadata
7. optionally prepare a biometric envelope for later unlock convenience
8. return the metadata payload to repository for persistence

The function should not write to the database directly.

Pseudo flow:

```text
master password
  -> Argon2id(salt, params)
  -> master-derived key
  -> wrap(vault key)
  -> vault_meta row
```

## Unlock Flow

Recommended `unlock()` flow:

1. load `VaultMetaRecord`
2. base64-decode salt and wrapped vault key
3. derive master-derived key from entered password using stored KDF params
4. unwrap vault key
5. if unwrap succeeds, cache vault key in memory
6. mark service unlocked
7. return `true`

If unwrap fails:

- clear any partial session state
- return `false`
- avoid leaking whether the failure came from password mismatch or malformed state

Pseudo flow:

```text
entered password
  -> Argon2id(stored salt, stored params)
  -> master-derived key
  -> unwrap(stored wrapped vault key)
  -> in-memory vault key
```

## Lock Flow

`lock()` should:

- zero or overwrite in-memory key buffers where possible
- clear any in-memory session object
- clear derived temporary values
- set unlocked state to false

Realistic note for Dart:

- Dart does not guarantee secure zeroization the same way low-level languages do
- still clear references promptly and avoid copying secret values more than necessary

## Biometric Unlock Strategy

For MVP, keep biometric unlock simple and explicit.

Recommended approach:

- after successful master-password unlock, optionally store a biometric-wrapped unlock envelope in platform secure storage
- the biometric prompt unlocks access to that envelope
- the envelope should contain enough material to restore vault access without storing the master password itself

Better approach:

- store a wrapped vault key protected by secure platform storage and gated behind biometrics

Avoid:

- storing the master password for biometric replay
- pretending biometrics replaces the master password cryptographically

The master password remains the root recovery secret.

## Encrypted Field Format

Each encrypted field should be stored as a self-contained binary envelope.

Recommended payload structure:

```text
version | algorithm_id | nonce_length | nonce | ciphertext
```

You can also serialize as JSON or protobuf, but for MVP a compact binary blob is better.

Suggested properties:

- `version`: helps future migration
- `algorithm_id`: supports future algorithm changes
- `nonce`: unique per field encryption
- `ciphertext`: includes auth tag if your crypto library appends it

Important:

- every encryption call must use a fresh random nonce
- never reuse a nonce with the same key

## Field Encryption Rules

Encrypt before writing to database:

- username
- password
- notes
- TOTP secret
- recovery codes

Can remain plaintext in MVP:

- title
- normalized domain
- created and updated timestamps
- favorite flag
- category

This matches the balanced MVP model used in the database design.

## Session State Design

Recommended internal session state:

```dart
class CryptoSession {
  CryptoSession({
    required this.vaultKey,
    required this.unlockedAt,
  });

  final List<int> vaultKey;
  final DateTime unlockedAt;
}
```

`CryptoService` can then hold:

```dart
CryptoSession? _session;
```

Expose:

- `isUnlocked => _session != null`

Do not expose:

- raw vault key
- derived master key

## Error Model

Recommended failure types:

```dart
class CryptoException implements Exception {
  CryptoException(this.message);

  final String message;
}

class VaultLockedException extends CryptoException {
  VaultLockedException() : super('Vault is locked');
}

class InvalidMasterPasswordException extends CryptoException {
  InvalidMasterPasswordException() : super('Invalid master password');
}

class CorruptCiphertextException extends CryptoException {
  CorruptCiphertextException() : super('Encrypted payload is invalid');
}
```

Guidance:

- repositories may translate these into domain-friendly failures
- UI should not differentiate too much between crypto corruption and invalid password during unlock

## Suggested Class Split

Do not let `CryptoService` grow into a giant file. Split by responsibility.

Recommended split:

- `crypto_service.dart`
- `key_derivation_service.dart`
- `cipher_service.dart`
- `payload_codec.dart`
- `secure_random.dart`

Suggested responsibility breakdown:

- `CryptoService`: orchestration and session state
- `KeyDerivationService`: Argon2id logic
- `CipherService`: encrypt and decrypt raw bytes
- `PayloadCodec`: serialize and parse encrypted payloads
- `SecureRandom`: nonce and key generation

## Example Service Shape

```dart
class CryptoServiceImpl implements CryptoService {
  CryptoServiceImpl({
    required KeyDerivationService keyDerivationService,
    required CipherService cipherService,
    required PayloadCodec payloadCodec,
    required SecureRandomService secureRandomService,
    required SecureKeyStorage secureKeyStorage,
  })  : _keyDerivationService = keyDerivationService,
        _cipherService = cipherService,
        _payloadCodec = payloadCodec,
        _secureRandomService = secureRandomService,
        _secureKeyStorage = secureKeyStorage;

  final KeyDerivationService _keyDerivationService;
  final CipherService _cipherService;
  final PayloadCodec _payloadCodec;
  final SecureRandomService _secureRandomService;
  final SecureKeyStorage _secureKeyStorage;

  CryptoSession? _session;

  @override
  bool get isUnlocked => _session != null;

  @override
  Future<VaultSetupResult> createVaultSetup(String masterPassword) async {
    throw UnimplementedError();
  }

  @override
  Future<bool> unlock({
    required String masterPassword,
    required VaultMetaRecord vaultMeta,
  }) async {
    throw UnimplementedError();
  }

  @override
  Future<List<int>> encryptToBytes(String plaintext) async {
    final session = _session;
    if (session == null) throw VaultLockedException();

    final nonce = _secureRandomService.randomBytes(12);
    final plaintextBytes = utf8.encode(plaintext);
    final encrypted = await _cipherService.encrypt(
      key: session.vaultKey,
      nonce: nonce,
      plaintext: plaintextBytes,
    );

    return _payloadCodec.encode(
      EncryptedPayload(
        version: 1,
        algorithm: 'aes-256-gcm',
        nonce: nonce,
        ciphertext: encrypted,
      ),
    );
  }

  @override
  Future<String> decryptFromBytes(List<int> ciphertext) async {
    final session = _session;
    if (session == null) throw VaultLockedException();

    final payload = _payloadCodec.decode(ciphertext);
    final plaintextBytes = await _cipherService.decrypt(
      key: session.vaultKey,
      nonce: payload.nonce,
      ciphertext: payload.ciphertext,
    );

    return utf8.decode(plaintextBytes);
  }

  @override
  Future<void> lock() async {
    _session = null;
  }

  @override
  Future<bool> canUnlockWithBiometricEnvelope() async {
    final value = await _secureKeyStorage.readBiometricEnvelope();
    return value != null;
  }

  @override
  Future<bool> unlockWithBiometricEnvelope() async {
    throw UnimplementedError();
  }
}
```

## KDF Parameter Guidance

Do not hardcode weak defaults without revisiting them.

Store these in `vault_meta`:

- KDF algorithm
- memory cost
- iteration count
- parallelism
- salt

Why:

- unlock must exactly reproduce the same derived key
- parameters may need tuning by device class or future app versions

For MVP:

- pick one tested parameter set
- store it per vault
- do not silently change it for existing vaults

## Database Interaction Contract

`CryptoService` should not fetch from `sqflite` directly.

Preferred design:

- repositories load metadata rows
- repositories pass normalized DTOs into `CryptoService`
- `CryptoService` returns encrypted blobs or unlock results

This separation matters because:

- it makes unit testing much easier
- it avoids mixing persistence concerns with crypto concerns
- it keeps Flutter app state simpler

## Memory Safety Notes

Within Dart and Flutter, you cannot get perfect memory secrecy. Still, you should:

- avoid logging secrets
- avoid keeping plaintext in provider state
- avoid copying secret values across multiple layers
- decrypt as late as possible
- re-lock on app background or timeout

Practical rule:

- plaintext should only exist briefly inside repository or page-level operations

## What Not To Do

- do not store plaintext passwords in `sqflite`
- do not store the master password in secure storage
- do not use one encryption nonce globally
- do not let UI screens call low-level crypto primitives directly
- do not bury unlock session state inside widgets
- do not make `CryptoService` query SQLite directly

## MVP Implementation Sequence

Recommended order:

1. implement key derivation
2. implement raw encrypt/decrypt
3. implement payload codec
4. implement `createVaultSetup()`
5. implement `unlock()`
6. implement `encryptToBytes()` and `decryptFromBytes()`
7. implement `lock()`
8. add biometric envelope path
