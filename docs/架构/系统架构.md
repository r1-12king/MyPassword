# 密码管理器系统架构文档

## 1. 文档目标

本文档用于定义当前密码管理器 MVP 的整体系统架构，基于已经确定的技术选型，统一以下内容：

- 技术栈边界
- 分层结构
- 模块职责
- 核心数据流
- 安全边界
- 后续扩展方向

这份文档是总纲，和已有设计文档的关系是：

- 本文档回答“整体怎么搭”
- `Flutter与Sqflite启动骨架.md` 回答“项目怎么启动”
- `本地数据库结构草稿.md` 回答“本地表怎么设计”
- `加密服务设计草稿.md` 回答“加密服务怎么设计”
- `安全密钥存储设计草稿.md` 回答“安全存储怎么设计”
- `保险库仓储设计草稿.md` 回答“保险库生命周期怎么编排”

## 2. 技术选型

当前推荐技术栈如下：

### 客户端框架

- `Flutter`
- `Dart`

原因：

- 一套代码覆盖 Android / iOS，后续也可扩展到桌面
- UI 迭代快
- 适合个人或小团队快速推进 MVP

### 状态管理

- `flutter_riverpod`

原因：

- 依赖注入和状态管理可以统一
- 测试和替换依赖更方便
- 比直接把状态散落在页面里更稳

### 路由

- `go_router`

原因：

- 路由结构清晰
- 适合初始化跳转、锁屏跳转、设置页流转这类场景

### 本地数据库

- `sqflite`

原因：

- 成熟稳定
- SQLite 行为可控
- 对 MVP 足够

注意：

- `sqflite` 只解决存储，不解决安全
- 敏感字段必须在写入前加密

### 安全存储

- `flutter_secure_storage`

原因：

- 用于保存少量敏感材料
- 适合存储生物识别解锁相关 envelope
- 比把这些值放进 SQLite 更合理

### 生物识别

- `local_auth`

原因：

- 满足 MVP 的指纹 / 面容解锁
- 可以和 `SecureKeyStorage` 配合实现生物识别解锁

### 密码学能力

推荐算法方向：

- KDF: `Argon2id`
- 字段加密: `AES-256-GCM` 或 `XChaCha20-Poly1305`

注意：

- 密码学实现不要自己发明
- 需要依赖成熟库或平台能力

## 3. 总体架构原则

本项目采用以下架构原则：

1. `本地优先`
   
   - MVP 阶段不依赖云服务
   - 所有核心功能先在本地闭环

2. `安全优先`
   
   - 明文密码不落盘
   - 主密码不持久化
   - vault key 只在解锁后存在于内存中

3. `职责分离`
   
   - UI 不直接碰数据库
   - Repository 不直接实现低层密码学
   - CryptoService 不直接写 SQLite

4. `先完成闭环，再做扩展`
   
   - 先做创建保险库、解锁、保存密码、查看密码
   - 后面再做自动填充、Passkey、同步

5. `面向后续扩展`
   
   - 当前架构要允许后面增加：
   - 云同步
   - TOTP
   - Passkey
   - 桌面支持

## 4. 分层架构

推荐采用 6 层结构：

```text
UI Layer
  ↓
Application / State Layer
  ↓
Domain Layer
  ↓
Repository Layer
  ↓
Data Layer
  ↓
Security Layer
```

说明：

- `UI Layer` 负责页面与交互
- `Application / State Layer` 负责页面状态和事件
- `Domain Layer` 负责接口与业务边界
- `Repository Layer` 负责业务编排
- `Data Layer` 负责本地数据库
- `Security Layer` 负责加密、密钥、安全存储、生物识别

虽然逻辑上是 6 层，但代码里不一定要做得很重。MVP 阶段重点是边界清晰，不是教科书式架构堆叠。

## 5. 模块划分

建议模块结构如下：

```text
lib/
  app/
  bootstrap/
  core/
  data/
  domain/
  security/
  features/
  shared/
```

### 5.1 `app/`

职责：

- 应用入口配置
- 路由注册
- 主题
- 全局 App 容器

典型内容：

- `app.dart`
- `router/app_router.dart`
- `router/route_names.dart`

### 5.2 `bootstrap/`

职责：

- 应用初始化
- 打开数据库
- 创建 DAO / Repository / Service
- 通过 Riverpod 注入依赖

这层是应用的组合根。

### 5.3 `core/`

职责：

- 常量
- 通用错误定义
- 基础工具方法

注意：

- `core/` 不要承载密码学实现
- 安全相关逻辑应单独放 `security/`

### 5.4 `data/`

职责：

- `sqflite` 初始化
- 数据库迁移
- DAO
- 数据模型
- Repository 实现依赖的数据访问层

这层只管“怎么存”和“怎么取”。

### 5.5 `domain/`

职责：

- Repository 抽象接口
- 领域实体
- 用例边界

这层是为了让 UI 和底层实现解耦。

### 5.6 `security/`

职责：

- `CryptoService`
- `SecureKeyStorage`
- `BiometricService`
- 密钥派生
- 加密与解密
- 安全随机数

这是本项目最不能和其他层混写的部分。

### 5.7 `features/`

职责：

- 按功能组织页面和状态

推荐功能目录：

- `lock`
- `setup`
- `home`
- `credential`
- `generator`
- `settings`
- `import_export`

### 5.8 `shared/`

职责：

- 公共组件
- 全局 Provider
- 可复用 UI 片段

## 6. 核心依赖关系

推荐依赖方向如下：

```text
features -> domain -> repositories -> data
                    -> security

bootstrap -> data
bootstrap -> security
bootstrap -> repositories
bootstrap -> app
```

更具体一点：

- 页面通过 Provider 调用 Repository
- Repository 调用 DAO 和 Security Service
- DAO 不依赖 UI
- CryptoService 不依赖页面和路由
- SecureKeyStorage 不依赖数据库

不允许的方向：

- 页面直接调用 DAO
- DAO 调用 CryptoService
- CryptoService 自己查 SQLite
- 页面直接操作 secure storage

## 7. 核心组件职责

### 7.1 `VaultRepository`

职责：

- 检查保险库是否存在
- 创建保险库
- 主密码解锁
- 生物识别解锁
- 锁定保险库
- 重置保险库

它是“保险库生命周期编排层”。

### 7.2 `CredentialRepository`

职责：

- 新增凭据
- 编辑凭据
- 查询凭据列表
- 获取凭据详情
- 删除凭据
- 搜索凭据

它负责把：

- 数据库存取
- 字段加解密
- 领域模型映射

组织成上层能直接调用的业务操作。

### 7.3 `CryptoService`

职责：

- 主密码派生密钥
- 生成 vault key
- wrap / unwrap vault key
- 维护解锁会话
- 字段加密和解密

它不负责：

- UI
- SQLite 查询
- 页面逻辑

### 7.4 `SecureKeyStorage`

职责：

- 存储生物识别解锁 envelope
- 存储少量安全敏感材料
- 删除相关安全材料

它不应该变成杂物箱。

### 7.5 `BiometricService`

职责：

- 判断设备是否支持生物识别
- 触发生物识别认证

它不负责：

- 解密
- 读写数据库
- 决定保险库业务状态

### 7.6 `DAO`

职责：

- 直接和 `sqflite` 交互
- 执行 SQL
- 返回原始记录

DAO 不应该知道：

- 主密码
- vault key
- UI 状态

## 8. 核心数据流

### 8.1 应用启动流

```text
main.dart
  -> AppBootstrap.initialize()
  -> open sqflite database
  -> create DAO / SecureKeyStorage / CryptoService / Repository
  -> ProviderScope
  -> GoRouter
  -> lock or setup page
```

### 8.2 创建保险库流

```text
SetupPage
  -> VaultRepository.createVault(masterPassword)
  -> CryptoService.createVaultSetup()
  -> VaultMetaDao.insertVaultMeta()
  -> optional SecureKeyStorage.writeBiometricEnvelope()
  -> go home
```

### 8.3 主密码解锁流

```text
LockPage
  -> VaultRepository.unlockVault(masterPassword)
  -> VaultMetaDao.getVaultMeta()
  -> CryptoService.unlock(masterPassword, vaultMeta)
  -> in-memory session established
  -> go home
```

### 8.4 生物识别解锁流

```text
LockPage
  -> BiometricService.authenticate()
  -> VaultRepository.unlockWithBiometrics()
  -> SecureKeyStorage.readBiometricEnvelope()
  -> CryptoService.unlockWithBiometricEnvelope(envelope)
  -> go home
```

### 8.5 保存凭据流

```text
CredentialEditPage
  -> CredentialRepository.saveCredential()
  -> CryptoService.encryptToBytes()
  -> CredentialDao.insert()
```

### 8.6 读取凭据详情流

```text
CredentialDetailPage
  -> CredentialRepository.getCredentialDetail()
  -> CredentialDao.getById()
  -> CryptoService.decryptFromBytes()
  -> UI render
```

## 9. 安全边界

### 9.1 主密码

规则：

- 只在用户输入时出现
- 不写入 SQLite
- 不写入 `flutter_secure_storage`
- 不进入日志
- 不进入路由参数

### 9.2 Vault Key

规则：

- 创建保险库时生成
- 平时以 wrapped 形式存储
- 解锁后存在内存会话中
- 锁定后立即清除引用

### 9.3 敏感字段

必须密文存储：

- 用户名
- 密码
- 备注
- TOTP secret
- 恢复码

可以明文存储的 MVP 字段：

- 标题
- 域名
- 创建时间
- 更新时间
- 收藏标记

这是为了兼顾本地搜索体验和 MVP 复杂度。

### 9.4 安全存储

建议只放：

- 生物识别解锁 envelope
- 少量会话辅助材料

绝不能放：

- 主密码
- 明文 vault key
- 明文密码记录

## 10. 目录与分层对应关系

```text
lib/app                -> 应用入口与路由层
lib/bootstrap          -> 启动装配层
lib/core               -> 基础支撑层
lib/domain             -> 业务边界层
lib/data               -> 数据访问层
lib/security           -> 安全能力层
lib/features           -> 页面与状态层
lib/shared             -> 共享组件与 Provider
```

## 11. 推荐的 Provider 组织方式

推荐分三类：

1. `基础依赖 Provider`
   
   - 数据库
   - DAO
   - SecureKeyStorage
   - CryptoService

2. `Repository Provider`
   
   - VaultRepository
   - CredentialRepository

3. `页面状态 Provider`
   
   - lockProvider
   - setupProvider
   - homeProvider
   - credentialEditProvider

原则：

- 页面只依赖页面状态 Provider 或 Repository Provider
- 页面不要直接依赖 DAO

## 12. 错误处理策略

推荐错误分层：

- `Data error`
  
  - 数据库失败
  - 记录不存在

- `Security error`
  
  - 解密失败
  - secure storage 读取失败
  - vault 已锁定

- `Domain error`
  
  - 保险库不存在
  - 主密码错误
  - 生物识别不可用

UI 层展示时要做收敛，不要把底层报错原样扔给用户。

## 13. MVP 范围内的架构取舍

当前架构明确做以下取舍：

### 做

- 单机本地保险库
- 单保险库模型
- 主密码解锁
- 生物识别解锁
- 本地加密字段
- 本地数据库

### 暂不做

- 多账号体系
- 云同步
- Passkey
- 浏览器扩展
- 自动填充服务
- 团队共享

这是为了把架构复杂度压在可控范围内。

## 15. 架构风险点

当前最需要注意的风险有：

1. `CryptoService` 过度膨胀
   
   - 解决方式：按 KDF、cipher、payload codec 拆分

2. `VaultRepository` 过度膨胀
   
   - 解决方式：只管 vault 生命周期，不接管凭据 CRUD

3. 页面直接访问 DAO
   
   - 解决方式：严格通过 Repository

4. Secure storage 被滥用
   
   - 解决方式：只存少量安全敏感材料

5. Provider 里混入过多业务逻辑
   
   - 解决方式：Provider 负责状态，业务逻辑放 Repository / Service

## 16. 推荐落地顺序

建议按这个顺序推进：

1. `bootstrap + router`
2. `sqflite database + vault_meta + credentials`
3. `CryptoService`
4. `SecureKeyStorage`
5. `VaultRepository`
6. `CredentialRepository`
7. `setup / lock / home / edit` 页面
8. `biometric unlock`
9. `import/export`

## 17. 结论

当前最合适的架构方案是：

- `Flutter` 作为跨平台 UI 容器
- `Riverpod` 作为依赖注入与状态承载
- `go_router` 作为导航层
- `sqflite` 作为本地数据库
- `flutter_secure_storage` 作为安全材料存储
- `CryptoService` 作为加密核心
- `VaultRepository` 作为保险库生命周期编排层
- `CredentialRepository` 作为凭据业务入口

这套架构的目标不是“看起来复杂”，而是：

- 先把本地密码库的安全闭环做对
- 让 MVP 能稳定实现
- 为后面做自动填充、同步、Passkey 留出扩展空间
