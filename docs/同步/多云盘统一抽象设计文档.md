# 多云盘统一抽象设计文档

本文档用于定义 `MyPassword` 的多云盘同步统一抽象方案。

与现有文档的区别：

- [多云盘同步模块设计.md](/Users/admin/Documents/mywork/cccli/MyPassword/docs/同步/多云盘同步模块设计.md)
  更偏产品方案和阶段规划
- 本文档
  更偏工程抽象、接口设计、模块边界、扩展策略

目标是解决一个核心问题：

- 第一阶段接入坚果云 WebDAV
- 第二阶段接入百度网盘
- 后续还可能接入其他网盘

如何保证：

- 上层业务流程不需要为每一种云盘重写一遍
- 页面层不需要知道每个云盘的协议差异
- 同步包格式保持统一
- 自动同步策略保持统一

## 1. 设计目标

多云盘统一抽象需要满足以下目标：

### 1. 统一同步对象

所有云盘都只处理：

- 加密后的同步包

而不是：

- SQLite 原始数据库文件
- 明文业务数据

### 2. 统一业务流程

无论是 WebDAV 还是百度网盘，上层业务都应该复用同一套流程：

- 测试连接 / 校验授权
- 上传同步包
- 下载同步包
- 读取远端元数据
- 恢复同步包

### 3. 统一状态与错误模型

页面层不应该关心：

- WebDAV 的 `401 / 403 / 404`
- 百度网盘的错误码
- 未来其他云盘的协议细节

页面层只关心：

- 认证失败
- 网络失败
- 远端文件不存在
- 配置无效
- 备份密码错误

### 4. Provider 可插拔

后续新增任何云盘时，理想状态应是：

1. 新增一个 Provider 实现
2. 新增对应配置模型
3. 注册进 Provider Registry
4. 上层同步流程无需改动

## 3. 推荐总体架构

建议统一架构如下：

```text
UI Layer
  Sync Settings
  Provider Settings Page
  Manual Upload / Restore Actions
  Sync Status Page
      ↓
Application Layer
  CloudSyncCoordinator
  CloudSyncAutomationService
  CloudSyncConfigService
      ↓
Domain Layer
  CloudSyncProvider
  CloudSyncProviderFactory / Registry
  CloudSyncModels
  CloudSyncException
      ↓
Data Layer
  CloudSyncRepository
  Local Sync State Storage
  Secure Provider Secret Storage
      ↓
Provider Layer
  WebDavSyncProvider
  BaiduPanSyncProvider
  Future providers...
      ↓
Package Layer
  exportBackup(...)
  importBackup(...)
```

设计原则：

- 页面层不直接碰 HTTP / OAuth / WebDAV
- 协调层不直接关心某个云盘的协议细节
- Provider 层不关心凭据条目结构
- 同步包始终复用本地备份格式

## 4. 统一抽象边界

### 4.1 Provider 层接口

当前项目已经有：

```dart
abstract class CloudSyncProvider {
  CloudSyncProviderType get providerType;
  String get displayName;

  Future<bool> isConfigured();
  Future<void> validateConnection();
  Future<RemoteSyncPackageMeta?> getRemoteMeta();
  Future<void> uploadPackage(SyncPackageUpload upload);
  Future<DownloadedSyncPackage?> downloadPackage(String path);
  Future<void> deletePackage(String path);
}
```

这个接口已经足够做第一阶段 WebDAV。

如果未来接百度网盘，建议保持这个接口不变，只在 Provider 内部适配百度网盘的 SDK / HTTP API。

### 4.2 Coordinator 层接口

当前协调层接口是：

```dart
abstract class CloudSyncCoordinator {
  Future<SyncStatusSnapshot> getStatus();
  Future<void> validateConnection();
  Future<RemoteSyncPackageMeta?> fetchRemoteMeta();
  Future<void> uploadCurrentVault({
    required String currentMasterPassword,
  });
  Future<DownloadedSyncPackage?> downloadLatestPackage();
}
```

这个设计很好，原因是：

- 页面不需要关心如何导出同步包
- 页面不需要关心如何调用 Provider
- 页面不需要关心同步状态如何记录

未来即使换成百度网盘，页面调用仍然可以保持不变。

### 4.3 Repository 层职责

当前 `CloudSyncRepository` 负责：

- 保存当前启用的 Provider 类型
- 保存 Provider 配置
- 保存最近本地变更时间
- 保存最近上传时间
- 保存最近下载时间
- 保存最近错误

建议未来也保持这个边界，不要让 Provider 自己去直接写 `SharedPreferences`。

## 5. 配置模型统一设计

当前配置模型为：

```dart
class CloudSyncConfig {
  final CloudSyncProviderType providerType;
  final String remotePath;
  final WebDavConfig? webDavConfig;
}
```

这适用于当前只有一个 WebDAV Provider 的阶段。

但如果未来接入百度网盘，建议升级为更清晰的扩展模型。

推荐形态：

```dart
class CloudSyncConfig {
  final CloudSyncProviderType providerType;
  final String remotePath;
  final WebDavConfig? webDavConfig;
  final BaiduPanConfig? baiduPanConfig;
}
```

或者进一步升级为：

```dart
abstract class CloudProviderConfig {
  CloudSyncProviderType get providerType;
}

class WebDavConfig implements CloudProviderConfig { ... }
class BaiduPanConfig implements CloudProviderConfig { ... }
```

推荐迁移策略：

- 当前阶段继续沿用现有结构
- 等第二个 Provider 接入时，再统一抽象 `CloudProviderConfig`

原因：

- 现阶段过度抽象收益不大
- 第二个 Provider 出现时，抽象边界会更清晰

## 6. Provider Registry 设计

当前项目已经有 Provider Registry：

```dart
final cloudSyncProviderRegistry = <CloudSyncProviderType, CloudSyncProvider>{
  CloudSyncProviderType.webdav: webDavProvider,
};
```

位于：

- [app_bootstrap.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/bootstrap/app_bootstrap.dart)

这是一个非常适合多云盘的基础设计。

未来新增 Provider 时，只需要：

```dart
final cloudSyncProviderRegistry = <CloudSyncProviderType, CloudSyncProvider>{
  CloudSyncProviderType.webdav: webDavProvider,
  CloudSyncProviderType.baiduPan: baiduPanProvider,
};
```

Coordinator 仍然通过：

- `providerType`
- `registry[providerType]`

来动态解析具体 Provider。

这就是标准的“Provider 可插拔架构”。

## 7. 同步包统一设计

多云盘统一抽象成立的前提，是所有云盘共享同一份同步包格式。

当前项目里已经这样做：

- 本地备份导出 -> 生成加密备份包
- WebDAV 上传 -> 上传该加密备份包
- WebDAV 恢复 -> 下载后走导入备份

因此未来百度网盘也应复用完全相同的同步包。

这样能保证：

- WebDAV 和百度网盘之间可以切换
- 用户从一个云盘迁移到另一个云盘时不需要转换格式
- 页面与业务层都不需要因为 Provider 变化而改同步包逻辑

统一原则：

- Provider 只处理“二进制同步包”
- Provider 不处理解密
- Provider 不知道条目结构

## 8. 错误模型统一设计

当前项目已经定义统一错误码：

- `notConfigured`
- `providerUnavailable`
- `authenticationFailed`
- `remoteFileNotFound`
- `networkError`
- `invalidConfiguration`
- `invalidMasterPassword`
- `invalidBackupPassword`
- `unknown`

这套错误模型非常适合保留为多云盘统一错误抽象。

原因：

- WebDAV 和百度网盘的底层错误码一定不同
- 但上层业务想表达的失败类型其实非常相似

建议以后所有 Provider 都先把原始错误映射到统一异常：

```dart
CloudSyncException(CloudSyncExceptionCode.xxx)
```

页面层永远只处理 `CloudSyncExceptionCode`。

## 9. 自动同步统一策略

当前自动同步逻辑位于：

- [cloud_sync_automation_service.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/domain/cloud_sync/cloud_sync_automation_service.dart)

它负责：

- 本地变更后自动上传
- 检查远端是否有新版本
- 提示潜在冲突

这个层面非常适合保持“与 Provider 无关”。

也就是说，自动同步策略应该只依赖：

- `CloudSyncCoordinator`
- `SyncStatusSnapshot`
- 当前会话主密码
- 当前设置项

而不应该直接依赖：

- WebDAV 协议
- 百度网盘 SDK

这样后面切换 Provider 时，自动同步策略可以原样复用。

## 11. WebDAV 与百度网盘的差异隔离方式

多云盘抽象真正的关键，不是“接口写出来”，而是“协议差异能不能只留在 Provider 层”。

### WebDAV Provider 特点

- 用户名 + 应用密码
- 基于 HTTP Basic Auth
- 用 `PROPFIND` / `PUT` / `GET` / `DELETE` / `MKCOL`
- 配置简单

### 百度网盘 Provider 特点

- OAuth 授权
- Access Token / Refresh Token
- 专有 REST API
- 可能有文件上传分片、预创建、任务轮询

这两者差异非常大。

所以统一抽象的重点不是强行让它们“内部一样”，而是：

- 让它们在 Provider 外部表现一样

即统一暴露：

- 可否连接 / 可否授权
- 上传同步包
- 下载同步包
- 获取远端元数据

## 12. 推荐的 Provider 分层模式

每个具体 Provider 建议拆三层：

### 1. Provider Facade

直接实现 `CloudSyncProvider`，向上提供统一接口。

例如：

- `WebDavSyncProvider`
- `BaiduPanSyncProvider`

### 2. Provider Client

只负责和远端 API 通信。

例如：

- `WebDavHttpClient`
- `BaiduPanApiClient`

### 3. Provider Config Mapper / Token Store

负责：

- 读取配置
- 读取安全凭据
- 更新 Token

这样做的好处是：

- Provider 本身不会越来越臃肿
- 单元测试更容易写
- 不同云盘的复杂度可以在各自内部消化

## 13. 第二阶段百度网盘建议设计

虽然当前还没实现百度网盘，但为了多云盘统一抽象，建议先明确方向。

### 13.1 配置模型

百度网盘建议使用：

```dart
class BaiduPanConfig {
  final String appKey;
  final String accessToken;
  final String refreshToken;
  final DateTime? expiresAt;
  final String remotePath;
}
```

### 13.2 安全存储建议

建议存储拆分：

- `appKey` 可普通存储
- `accessToken / refreshToken` 安全存储

### 13.3 Provider 行为保持一致

虽然百度网盘可能需要更多中间步骤，但最终仍然向上暴露：

- `validateConnection()`
- `getRemoteMeta()`
- `uploadPackage(...)`
- `downloadPackage(...)`

上层完全不需要知道百度网盘上传是否经历了“预创建 / 分片 / 合并”等细节。

## 14. 迁移到其他工程时推荐步骤

如果你要把多云盘统一抽象迁移到其他项目，建议按下面顺序推进。

### 第一步：先定同步包格式

不要先写 Provider。

先确定：

- 本地备份格式
- 云同步包是否与本地备份共用
- 导入恢复流程是否复用本地导入

### 第二步：落统一接口

先定义：

- `CloudSyncProvider`
- `CloudSyncCoordinator`
- `CloudSyncRepository`
- `CloudSyncException`

### 第三步：先接一个最简单 Provider

建议第一阶段优先：

- WebDAV

因为它：

- 调试简单
- 协议公开
- 用户自己可控

### 第四步：补 Provider Registry

即使暂时只有一个 Provider，也先把 Registry 立起来。

### 第五步：再做自动同步

不要一开始就做自动同步。

建议顺序：

1. 手动上传
2. 手动恢复
3. 远端更新时间检查
4. 自动上传
5. 冲突提示

### 第六步：再接第二个 Provider

只有接入第二个 Provider 时，才能真正验证你的抽象是否合理。

## 16. 结论

多云盘统一抽象的核心，不是让所有云盘内部实现都一样，而是：

- 让所有云盘在业务层看来都一样

当前项目已经具备很好的第一阶段基础：

- 统一同步包
- 统一 Provider 接口
- 统一 Coordinator
- 统一错误模型
- 统一自动同步策略

后续只要沿着这个边界继续推进：

- WebDAV
- 百度网盘
- 其他云盘

都可以逐步接入，而不会把页面层和业务层拖乱。
