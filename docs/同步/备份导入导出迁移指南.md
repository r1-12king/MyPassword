# 备份导出 / 导入 / 自定义导出目录实现方案

本文档基于 `MyPassword` 当前实现，整理一套可迁移到其他 Flutter 工程的参考方案。

目标覆盖以下能力：

- 导出备份
- 导入备份
- 自定义导出目录
- Android 原生目录选择
- Android 原生文件选择
- 备份文件主密码校验
- 导入重复项跳过

本文档重点不是“页面长什么样”，而是“这一套能力技术上如何落地”。

---

## 1. 目标能力定义

当前项目中的能力定义如下：

### 1. 导出备份

- 只能在保险库已解锁时导出
- 导出前必须输入当前应用主密码
- 导出的不是明文数据库，而是“业务数据 JSON + 密码校验信息”组成的加密备份包
- 导出文件写入用户选定的目录

### 2. 导入备份

- 用户先选备份文件
- 用户输入“备份文件主密码”
- 先校验该主密码是否能正确解密当前备份文件
- 校验通过后才执行导入
- 如果当前设备还没有保险库，先要求用户创建当前设备的新保险库主密码
- 导入时保留当前保险库主密码不变，只把老备份内容导入进来

### 3. 自定义导出目录

- 用户在设置页先选择一次导出目录
- 目录 URI 和可展示名称持久化保存
- 后续每次导出都直接使用该目录
- 直到用户下次重新修改目录为止

## 3. 核心设计思路

这套实现方案的关键点有 4 个：

### 1. 不直接导出 SQLite 文件

项目没有直接把数据库文件原样拷出去，而是：

- 先把当前所有凭据读取成结构化 JSON
- 再把 JSON 使用“当前主密码”重新加密
- 最终写成一个应用自定义备份文件

优点：

- 备份格式可控
- 不依赖数据库文件结构的直接兼容
- 可以在备份中加入版本号、时间戳、密码校验信息
- 后续迁移数据库结构时更容易做版本升级

### 2. 备份文件密码与当前应用主密码绑定

当前实现中，导出备份时使用“当前应用主密码”作为备份包的加密密码。

也就是说：

- 导出时输入的是当前应用主密码
- 导入时要求输入的是当时导出该备份文件所用的主密码

这也是当前项目里“备份文件主密码”的来源。

### 3. 自定义导出目录不保存绝对文件路径，只保存目录 URI

在 Android 上，用户通过系统目录选择器选中的并不是普通文件系统路径，而是一个可持久授权的 `treeUri`。

因此正确做法不是：

- 保存 `/storage/emulated/0/...`

而是：

- 保存系统返回的目录 URI
- 下次写文件时继续用该 URI 操作

### 4. 导入文件先拷贝到缓存目录，再交给仓储层处理

导入备份不是直接让业务层读 `content://` URI，而是先在 Android 原生层：

- 通过 `ContentResolver.openInputStream(uri)` 打开用户选择的文件
- 拷贝到应用缓存目录生成普通文件路径
- 再把本地缓存文件路径返回给 Flutter

这样做的优点是：

- Flutter 数据层只需要处理普通文件路径
- 不需要让仓储层感知 Android `content://` URI
- 导入逻辑更容易迁移到其他项目

## 4. 自定义导出目录实现方法

```dart

```

返回值包含：

- `uri`
- `label`

这两个字段的作用：

- `uri` 用来真正写文件
- `label` 用来在设置页展示给用户看

### 4.2 Android 侧目录选择

原生实现使用：

- `Intent.ACTION_OPEN_DOCUMENT_TREE`

对应代码：

```kotlin
contentResolver.takePersistableUriPermission(uri, flags)
```

这样应用后续仍然可以访问这个目录，不需要每次重新授权。

### 4.3 导出目录持久化

设置时调用：

```dart
  required String uri,

```

### 4.4 设置页接入方式

设置页中点击“导出目录”时：

1. 调 `ExportDirectoryService.pickDirectory()`
2. 得到 `uri + label`
3. 调 `AppSettingsController.setExportDirectory(...)`

## 5. 导出备份实现方法

### 5.1 UI 层交互

导出页当前实现逻辑：

1. 用户输入当前主密码
2. 读取当前设置中的导出目录 URI
3. 调 `credentialRepository.exportBackup(...)`
4. 调 `exportDirectoryService.writeBackupFile(...)`
5. 展示返回的输出路径或 URI

对应代码：

- [export_backup_page.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/features/export_import/presentation/pages/export_backup_page.dart)

页面层的关键校验：

- 主密码不能为空
- 导出目录必须已经设置

### 5.2 仓储层导出接口

导出接口定义：

```dart
Future<BackupExportData> exportBackup({
  required String currentMasterPassword,
})
```

对应代码：

- [credential_repository.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/domain/repositories/credential_repository.dart)

返回对象：

```dart
class BackupExportData {
  final String fileName;
  final List<int> bytes;
}
```

这样设计的好处是：

- 仓储层只负责生成导出文件内容
- 不负责决定最终写到哪个目录
- 文件写入交给平台层处理

这是一种非常适合迁移的分层方案。

### 5.3 导出前主密码校验

导出逻辑首先会做两层校验：

1. 保险库必须已解锁
2. 当前输入的主密码必须与当前保险库主密码匹配

对应代码：

- [credential_repository_impl.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/data/repositories/credential_repository_impl.dart)

实际逻辑：

- 从 `vault_meta` 读取当前保险库元信息
- 使用 `CryptoService.verifyMasterPassword(...)` 校验输入主密码
- 校验失败直接抛错

### 5.4 导出数据结构

当前导出的 JSON 结构大致如下：

```json
{
  "version": 2,
  "exportedAt": 1710000000000,
  "masterPasswordVerifier": {
    "kdf": "pbkdf2_sha256",
    "iter": 120000,
    "salt": "...",
    "digest": "..."
  },
  "credentials": [
    {
      "id": "...",
      "title": "...",
      "username": "...",
      "password": "...",
      "websiteUrl": "...",
      "websiteDomain": "...",
      "notes": "...",
      "category": "...",
      "favorite": true,
      "createdAt": 1710000000000,
      "updatedAt": 1710000000000
    }
  ]
}
```

这里有三个关键字段：

- `version`
- `exportedAt`
- `masterPasswordVerifier`

其中 `masterPasswordVerifier` 非常关键，它不是主密码本身，而是“主密码校验信息”。

这样导入时就能做到：

- 仍然要求用户手动输入备份文件主密码
- 但可以程序化校验输入是否正确

这正符合当前项目的产品要求。

### 5.5 导出加密方式

导出 JSON 生成后，不是明文落盘，而是调用：

```dart
CryptoService.encryptWithPassword(...)
```

对应代码：

- [crypto_service.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/security/crypto/crypto_service.dart)

当前实现方案：

- 使用 PBKDF2-SHA256 从主密码派生密钥
- 每次生成新的随机 salt
- 用 AES-256-GCM 加密明文 JSON
- 最终输出一个 JSON envelope，再转成字节写盘

这层 envelope 大致如下：

```json
{
  "v": 1,
  "kdf": "pbkdf2_sha256",
  "iter": 120000,
  "salt": "...",
  "payload": "..."
}
```

因此最终备份文件实际上是：

- 外层：备份加密 envelope
- 内层：业务备份 JSON

## 6. 写入自定义导出目录实现方法

### 6.1 Flutter 侧写文件接口

Flutter 不直接使用 `dart:io` 写用户选择的公共目录，而是调用：

```dart
writeBackupFile({
  required String directoryUri,
  required String fileName,
  required Uint8List bytes,
})
```

对应代码：

- [export_directory_service.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/core/export/export_directory_service.dart)

### 6.2 Android 侧写文件实现

原生层通过 `DocumentFile.fromTreeUri(...)` 拿到目录对象，然后：

1. 检查目录是否可写
2. 如果同名文件存在，先删除
3. 创建新文件
4. 通过 `ContentResolver.openOutputStream(...)` 写入字节流

对应代码：

- [MainActivity.kt](/Users/admin/Documents/mywork/cccli/MyPassword/android/app/src/main/kotlin/com/example/my_password/MainActivity.kt)

关键代码点：

```kotlin
val directory = DocumentFile.fromTreeUri(this, treeUri)
directory.findFile(fileName)?.delete()
val document = directory.createFile("application/octet-stream", fileName)
contentResolver.openOutputStream(document.uri)?.use { stream ->
    stream.write(bytes)
    stream.flush()
}
```

### 6.3 为什么不用普通文件路径

因为 Android 10+ 以后，普通外部存储路径越来越不稳定，Scoped Storage 下直接写公共目录会受限制。

因此迁移到其他项目时，建议优先使用：

- Storage Access Framework
- `ACTION_OPEN_DOCUMENT_TREE`
- `DocumentFile`

而不是硬编码外部路径。

## 7. 导入备份实现方法

### 7.1 UI 层交互

导入页当前流程：

1. 点击“选择备份文件”
2. 调 `pickBackupFile()`
3. 获得缓存文件路径
4. 用户输入备份文件主密码
5. 如果当前没有保险库，再输入当前设备要创建的新主密码
6. 先调用 `verifyBackupPassword(...)`
7. 校验通过后，再调用 `importBackup(...)`

对应代码：

- [import_backup_page.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/features/export_import/presentation/pages/import_backup_page.dart)

### 7.2 为什么先校验再导入

导入前先独立执行：

```dart
verifyBackupPassword(filePath, backupMasterPassword: ...)
```

这样做的好处是：

- 主密码错误时可以单独提示“备份文件主密码错误”
- 不会把所有异常都折叠成“无法导入备份”
- 用户体验明显更好

### 7.3 Android 原生文件选择

原生层使用：

- `Intent.ACTION_OPEN_DOCUMENT`

对应代码：

- [MainActivity.kt](/Users/admin/Documents/mywork/cccli/MyPassword/android/app/src/main/kotlin/com/example/my_password/MainActivity.kt)

当前逻辑：

1. 系统文件选择器返回 `content://` URI
2. 原生层读入该文件
3. 拷贝到应用 `cacheDir`
4. 返回普通绝对路径给 Flutter

关键代码：

```kotlin
val targetFile = File(cacheDir, fileName)
contentResolver.openInputStream(uri)?.use { input ->
    targetFile.outputStream().use { output ->
        input.copyTo(output)
    }
}
```

迁移建议：

- 如果你希望 Flutter 业务层尽量跨平台，强烈建议采用这种“原生层转缓存文件”的策略

### 7.4 备份解密与密码校验

导入校验和导入正式执行都依赖 `_readBackupPayload(...)`。

对应代码：

- [credential_repository_impl.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/data/repositories/credential_repository_impl.dart)

其流程是：

1. `File(filePath).readAsBytes()`
2. `CryptoService.decryptWithPassword(...)`
3. `jsonDecode(...)`
4. 读取 `masterPasswordVerifier`
5. `CryptoService.verifyPasswordWithVerifier(...)`
6. 校验失败则抛出 `Invalid backup password`

这里的设计重点是：

- “能解密”不等于“密码正确”
- 还要再用 `masterPasswordVerifier` 做一次显式校验

这是当前项目里实现“备份文件主密码校验信息/标识”的核心方法。

### 7.5 当前无保险库时的处理

如果当前设备还没有保险库：

- 导入页会额外要求用户输入“当前设备的新主密码”
- 先创建一个新的本地保险库
- 再把老备份内容导入当前保险库

对应代码：

- [import_backup_page.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/features/export_import/presentation/pages/import_backup_page.dart)

这满足当前项目的产品要求：

- 保留当前保险库主密码体系
- 把老备份数据导入当前设备
- 不直接把设备主密码替换成老备份主密码

## 8. 导入重复项处理实现方法

规则是：

- 标题
- 用户名
- 域名

三者组合成唯一键：

```dart
title.trim().toLowerCase()
username?.trim().toLowerCase() ?? ''
websiteDomain?.trim().toLowerCase() ?? ''
```

对应代码：

- [credential_repository_impl.dart](/Users/admin/Documents/mywork/cccli/MyPassword/lib/data/repositories/credential_repository_impl.dart)

当前策略：

- 已存在则跳过
- 不覆盖
- 不弹冲突选择框

这是一个偏保守、低风险的第一阶段方案。

如果迁移到其他项目，可以把这里扩展成：

- 跳过
- 覆盖
- 另存副本
- 用户逐条决定

## 9. 迁移到其他项目时建议保留的抽象边界

如果你要把这套方案迁移到其他工程，建议保留下面几个接口边界，不要把它们耦合在一起。

### 1. 仓储层只负责“生成备份内容”和“解析备份内容”

不要让仓储层直接处理：

- 目录选择
- 系统文件选择器
- Android URI 权限

仓储层更适合只暴露：

```dart
exportBackup(...)
verifyBackupPassword(...)
importBackup(...)
```

### 2. 文件访问能力独立成平台服务

建议单独保留一个类似：

```dart
ExportDirectoryService
```

专门负责：

- pickDirectory
- pickBackupFile
- writeBackupFile

这样平台差异只集中在这一层。

### 3. 导出目录配置单独存到设置层

不要每次导出都临时让用户选目录。

建议像当前项目一样，把目录配置放在设置里，长期生效。

### 4. 备份文件格式要带版本号

建议在业务 payload 中始终保留：

- `version`
- `exportedAt`
- `masterPasswordVerifier`

这样未来格式升级、兼容老备份文件时会容易很多。

## 11. 当前实现的局限

当前项目实现虽然可用，但还存在一些明确边界：

### 1. 目前只实现了 Android 原生文件桥接

如果迁移到 iOS：

- 需要单独实现 iOS 文件选择和目录写入逻辑
- Flutter 侧接口可以保持不变

### 2. 导出文件仍依赖主密码

当前方案要求：

- 自动导出或自动同步时，需要拿到当前主密码或等价授权能力

这也是为什么当前项目里自动同步在生物识别解锁场景下不能直接自动上传。

### 3. 重复项策略较简单

当前只支持：

- 识别重复
- 自动跳过

还没有做冲突处理 UI。

## 12. 结论

这套实现方案适合以下类型项目：

- 本地优先的密码管理器
- 本地优先的私密笔记
- 本地优先的加密数据管理 App

它的核心优势是：

- 备份格式独立于数据库文件
- 主密码校验逻辑清晰
- 平台文件访问与业务导入导出解耦
- 可扩展到云同步场景
