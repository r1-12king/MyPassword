# 多云盘同步模块设计

## 1. 文档目标

本文档用于定义密码管理器的多云盘同步模块设计，目标满足以下产品前提：

- 用户自己决定是否启用同步
- 用户自己授权并使用自己的个人云盘账号
- App 只负责上传和下载用户自己的加密同步包
- 服务端不托管用户数据
- 第一阶段优先接入 `坚果云 WebDAV`
- 第二阶段预留 `百度网盘`
- 后续允许继续扩展其他云盘

本文档回答以下问题：

- 多云盘同步模块怎么分层
- 同步包格式怎么定义
- Provider 如何抽象
- 本地数据库和设置项要增加什么
- 一期 WebDAV 怎么落地
- 二期百度网盘怎么接入

---

## 2. 设计原则

### 2.1 本地优先

- 本地 SQLite 仍然是主数据源
- 云端仅保存加密同步包
- 不直接把 SQLite 原始数据库文件作为同步对象

### 2.2 Provider 解耦

- 同步包格式与云盘类型解耦
- 上传下载流程与具体 Provider 解耦
- WebDAV、百度网盘、阿里云盘都只作为适配层实现

### 2.3 安全边界清晰

- 云端只存加密包
- 用户主密码不明文上传
- 恢复时由 App 自动处理解密，但仍要求用户输入主密码

### 2.4 先做手动同步，再做自动同步

一期先做：

- 绑定云盘
- 手动上传同步包
- 手动下载同步包
- 版本检查和覆盖确认

后续再做：

- 修改后提醒同步
- 自动同步
- 多设备冲突管理

---

## 3. 总体架构

推荐结构：

```text
UI Layer
  Settings / Sync Screens
      ↓
Application Layer
  CloudSyncCoordinator
  SyncStateController
      ↓
Domain Layer
  CloudSyncProvider
  SyncPackageBuilder
  SyncConflictPolicy
      ↓
Data Layer
  SyncMetadataDao
  SyncSettingsStorage
      ↓
Security Layer
  CryptoService
      ↓
Provider Layer
  WebDavProvider
  BaiduPanProvider
  Future providers...
```

说明：

- `UI Layer` 只负责用户发起同步、恢复、查看状态
- `Coordinator` 负责统一同步编排
- `Provider` 只负责具体云盘的授权、上传、下载、列元数据
- `CryptoService` 负责同步包加密和解密

---

## 4. 核心模块设计

## 4.1 `CloudSyncProvider`

这是所有云盘接入的统一抽象。

建议接口：

```dart
abstract class CloudSyncProvider {
  String get providerId;
  String get displayName;

  Future<bool> isAuthorized();
  Future<void> authorize();
  Future<void> revoke();

  Future<RemoteSyncPackageMeta?> getRemoteMeta();
  Future<void> uploadPackage(SyncPackageUpload upload);
  Future<DownloadedSyncPackage?> downloadPackage();
}
```

### 统一要求

- Provider 不感知业务密码条目结构
- Provider 不负责加密明文
- Provider 只处理“已经加密好的同步包”

---

## 4.2 `CloudSyncCoordinator`

这是同步总编排器。

职责：

- 读取当前启用的 Provider
- 校验是否已授权
- 构建同步包
- 调用上传
- 下载远端同步包
- 检查本地/远端版本差异
- 触发恢复导入流程

建议职责拆分：

- `uploadNow()`
- `checkRemoteUpdate()`
- `downloadAndRestore()`
- `compareVersion()`

---

## 4.3 `SyncPackageBuilder`

职责：

- 从当前库中导出业务数据
- 生成稳定的同步包结构
- 调用 `CryptoService` 加密

注意：

- 不同步 SQLite 原始文件
- 只同步应用定义的标准格式

---

## 4.4 `SyncRestoreService`

职责：

- 下载同步包
- 读取元数据
- 校验主密码
- 解密同步包
- 导入本地数据库

导入策略：

- 当前应用主密码保持不变
- 远端同步包主密码仅用于解密远端包
- 导入后仍以本地保险库体系运行

---

## 4.5 `SyncConflictPolicy`

一期建议先做最简单可控策略：

- 文件级版本比较
- 不做自动记录级合并
- 如果远端版本更新：
  - 提示用户是否恢复远端
- 如果本地版本更新：
  - 提示用户是否上传覆盖远端

后续再扩展为：

- 记录级版本
- 设备级变更日志
- 自动冲突合并

---

## 5. 同步包格式设计

建议扩展名：

- `vault_sync.mpsync`

建议结构：

```json
{
  "version": 1,
  "exportedAt": 1710000000000,
  "deviceId": "device-uuid",
  "appVersion": "0.2.0",
  "providerHint": "webdav",
  "masterPasswordVerifier": {
    "kdf": "pbkdf2_sha256",
    "iter": 120000,
    "salt": "...",
    "digest": "..."
  },
  "payloadHash": "...",
  "encryptedPayload": "..."
}
```

其中：

- `masterPasswordVerifier` 只用于校验主密码，不保存主密码本身
- `encryptedPayload` 为真正业务数据的加密体
- `payloadHash` 用于完整性检查

业务数据结构建议：

```json
{
  "credentials": [...],
  "deletedRecords": [...],
  "meta": {
    "vaultVersion": 1,
    "schemaVersion": 1
  }
}
```

## 6. Provider 分阶段设计

## 6.1 WebDAV Provider

适用一期。

能力：

- 保存 WebDAV 服务地址
- 保存用户名
- 保存应用密码或第三方密码
- 检查远端同步文件是否存在
- 上传同步包
- 下载同步包
- 获取远端修改时间和 etag

建议固定路径：

- `/MyPassword/vault_sync.mpsync`

优点：

- 协议标准
- 易于适配多个 WebDAV 服务
- 先接坚果云，后续理论上也兼容其他 WebDAV 服务

---

## 6.2 百度网盘 Provider

适用二期预研后接入。

能力要求：

- OAuth 授权
- AccessToken/RefreshToken 管理
- 文件上传
- 文件下载
- 文件元数据查询

风险：

- 个人盘开放能力可能存在权限边界和政策变动
- 接口稳定性需在预研阶段单独验证

设计建议：

- 独立 `BaiduPanProvider`
- 不影响一期 WebDAV 主路径
- 如果预研失败，不影响整体架构

---

## 7. 本地存储设计

建议新增本地设置项：

### `sync_settings`

建议字段：

- `provider_id`
- `enabled`
- `remote_path`
- `authorized_at`
- `last_sync_at`
- `last_upload_at`
- `last_download_at`
- `auto_sync_enabled`
- `sync_on_launch`

对于敏感信息：

- WebDAV 用户名可存在 SQLite 或 SharedPreferences
- WebDAV 应用密码必须放安全存储
- OAuth token 必须放安全存储

### `sync_state`

建议记录：

- `last_local_package_hash`
- `last_remote_package_hash`
- `last_remote_updated_at`
- `last_sync_result`
- `last_error_code`

## 9. 一期推荐同步流程

### 上传流程

1. 用户点击“同步到云端”
2. Coordinator 检查当前 Provider 是否已授权
3. 读取本地数据并生成同步包
4. 加密同步包
5. 上传到远端固定路径
6. 保存本地同步状态

### 恢复流程

1. 用户点击“从云端恢复”
2. Coordinator 获取远端元数据
3. 下载远端同步包
4. 用户输入主密码
5. App 自动校验并解密
6. 用户确认恢复
7. 导入本地库

---

## 10. 冲突与覆盖策略

一期明确采用以下策略：

- 不做自动合并
- 如果远端比本地新：
  - 不允许静默上传覆盖
- 如果本地比远端新：
  - 允许用户手动上传覆盖
- 如果时间一致但 hash 不一致：
  - 标记冲突，禁止自动处理

这是为了避免：

- 多设备同时修改导致数据丢失
- 用户误覆盖远端较新版本

---

## 11. 安全注意事项

- 不上传主密码明文
- 不上传本地解密后的 SQLite 文件
- Provider 层不接触业务明文
- 所有密钥校验和解密动作都在本地进行
- 远端同步文件应允许覆盖，但不能静默丢失历史信息

建议后续增强：

- 同步包完整性签名
- 设备绑定信息
- 多版本历史文件保留

---

## 12. 版本演进建议

### V1

- 单文件同步
- 单 Provider
- 手动上传/下载
- 主密码解密恢复

### V2

- 多 Provider
- 启动时检查远端更新
- 修改后提醒同步

### V3

- 自动同步
- 多设备冲突提示
- 版本历史

### V4

- 记录级同步
- 设备级变更日志
- 自动合并策略

---

## 13. 当前结论

对于当前项目，推荐路径是：

1. 先抽象 `CloudSyncProvider`
2. 一期只实现 `WebDavProvider`
3. 同步标准化的加密同步包
4. 二期单独预研百度网盘
5. 所有 Provider 都通过统一协调器接入

这条路线最稳，也最符合当前本地优先架构。
